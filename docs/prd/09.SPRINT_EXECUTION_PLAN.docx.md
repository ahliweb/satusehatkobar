# SPRINT\_EXECUTION\_PLAN.md

## Rencana Eksekusi Sprint MVP Satu Sehat Kobar v1.4

### Panduan Tahapan Implementasi Atomic untuk AI Coding Agent dan Developer

---

## 1\. Tujuan Dokumen

Dokumen ini menjadi panduan pelaksanaan sprint untuk implementasi MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro**.

Dokumen ini digunakan agar pelaksanaan pengembangan:

1. Berjalan bertahap.

2. Tidak melebar dari scope MVP.

3. Selalu mengikuti urutan dependensi teknis.

4. Dapat dikerjakan oleh AI coding agent secara atomic.

5. Dapat dilacak melalui GitHub issues.

6. Dapat divalidasi melalui acceptance criteria.

7. Dapat diuji sebelum lanjut ke sprint berikutnya.

8. Tetap menjaga batas core AWCMS-Micro.

9. Selalu memperbarui dokumentasi.

10. Siap masuk UAT dan pilot terbatas.

---

## 2\. Prinsip Eksekusi Sprint

Setiap sprint wajib mengikuti prinsip berikut:

1. **Satu sprint memiliki target terbatas.**

2. **Satu issue memiliki satu fokus pekerjaan.**

3. **Tidak boleh lanjut ke sprint berikutnya jika Must Have sprint aktif belum selesai.**

4. **Tidak boleh mengerjakan fitur Won’t Have MVP.**

5. **Tidak boleh mengubah core AWCMS-Micro tanpa keputusan teknis.**

6. **Setiap perubahan database harus melalui migration yang aman.**

7. **Setiap endpoint harus memiliki permission check.**

8. **Setiap aksi penting harus memiliki audit trail.**

9. **Setiap fitur utama harus memiliki validasi/test.**

10. **Setiap sprint selesai harus menghasilkan laporan sprint.**

---

## 3\. Alur Sprint Keseluruhan

Sprint 0

# 15\. Definition of Ready Sprint

Sebuah sprint siap dimulai bila:

* Sprint sebelumnya selesai.

* Gate sprint sebelumnya terpenuhi.

* Issue sprint sudah tersedia.

* Issue memiliki acceptance criteria.

* Dependensi diketahui.

* Scope sprint jelas.

* Di luar scope sprint jelas.

* Tidak ada blocker kritis.

* Environment dapat digunakan.

* Command validasi diketahui.

---

# 16\. Definition of Done Sprint

Sebuah sprint selesai bila:

* Semua issue Must Have selesai.

* Acceptance criteria issue terpenuhi.

* Test/validasi dijalankan.

* Dokumentasi diperbarui.

# 17\. Risiko Sprint dan Mitigasi

| Risiko | Mitigasi |
| :---- | :---- |
| Scope melebar | Gunakan MoSCoW dan Won’t Have MVP. |
| AI mengubah core | Terapkan plugin boundary dan stop condition. |
| Migration merusak data | Backup, staging test, migration kecil. |
| Permission kurang | Test role/ABAC setiap sprint. |
| PDF tidak sesuai format | Validasi dengan Sekretariat/Operator Surat. |
| Dashboard lambat | Gunakan aggregate endpoint dan index. |
| Upload file tidak aman | Validasi MIME, size, extension, permission. |
| User bingung | Wizard, label jelas, manual pengguna. |
| UAT terlambat | Libatkan user sejak Sprint 4–5 untuk preview. |
| Bug critical saat pilot | Rollback plan dan incident report. |
