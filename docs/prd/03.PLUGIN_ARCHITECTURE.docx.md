# PLUGIN\_ARCHITECTURE.md

## Arsitektur Plugin MVP Satu Sehat Kobar v1.4

### Berbasis AWCMS-Micro Modular Plugin Ecosystem

---

## 1\. Tujuan Dokumen

Dokumen ini menjelaskan arsitektur plugin untuk implementasi MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro**.

Dokumen ini digunakan oleh:

1. AI coding agent.

2. Tim SIK Dinas Kesehatan Kotawaringin Barat.

3. Developer plugin.

4. System architect.

5. QA/tester.

6. Admin teknis.

7. Reviewer keamanan.

8. Dokumentator sistem.

Tujuan utama dokumen ini adalah memastikan seluruh fitur Satu Sehat Kobar dikembangkan melalui pola **plugin ecosystem** yang rapi, aman, modular, dan tidak merusak core AWCMS-Micro.

---

## 2\. Prinsip Arsitektur Plugin

## 2.1 Prinsip Utama

1. **Core tetap ringan.** Core AWCMS-Micro tidak menjadi tempat menanam logika bisnis khusus Satu Sehat Kobar.

2. **Fitur daerah dikembangkan sebagai plugin.** Agenda, ST/SPPD, SPM, MMC, arsip, dan dashboard dikembangkan melalui plugin.

3. **Plugin saling berkomunikasi melalui service contract.** Plugin tidak boleh membaca tabel internal plugin lain secara langsung kecuali melalui kontrak resmi.

4. **Plugin harus dapat diuji secara terpisah.** Setiap plugin memiliki migration, service, route, UI, permission, audit trail, test, dan dokumentasi sendiri.

5. **Plugin harus aman secara akses.** Setiap route dan action plugin wajib memeriksa authentication, RBAC, dan ABAC.

6. **Plugin wajib audit-friendly.** Setiap aksi penting harus masuk audit trail.

7. **Plugin wajib release-friendly.** Perubahan plugin harus memiliki changelog dan dokumentasi.

8. **Plugin tidak boleh membuat scope MVP melebar.** Plugin MVP hanya fokus pada siklus Agenda → ST/SPPD → Approval → PDF → Bukti → Jurnal → Dashboard → Arsip.

1. Business logic.

2. Database migration khusus fitur.

4. UI halaman fitur.

5. Permission khusus fitur.

6. Audit event khusus fitur.

7. Template dokumen.

8. Data validation.

9. Report/export.

10. Dashboard widget.

AI coding agent dan developer dilarang:

1. Menanam logika Agenda ke core.

2. Menanam logika ST/SPPD ke core.

3. Menanam format ST/SPPD khusus Dinkes ke core.

4. Menanam workflow approval daerah ke core.

6. Membaca tabel internal plugin lain tanpa service contract.

7. Menghapus atau mengubah struktur core tanpa keputusan teknis.

8. Membuat plugin bergantung pada fitur Won’t Have MVP.

9. Membuat integrasi eksternal penuh pada MVP tanpa instruksi eksplisit.

## 7\. Plugin Registry

## 7.1 Tujuan Plugin Registry

Plugin registry digunakan untuk:

1. Mencatat plugin yang tersedia.

2. Mengaktifkan atau menonaktifkan plugin.

3. Menampilkan plugin pada admin UI.

4. Menentukan dependency antar-plugin.

5. Menyimpan metadata plugin.

6. Mendukung dashboard modular.

7. Mencegah penggunaan plugin yang belum siap.

8. Menjadi dasar audit perubahan status plugin.

2. Plugin tidak boleh crash bila plugin optional belum aktif.

3. Plugin harus memeriksa status dependency melalui plugin registry.

4. Dependency hard harus divalidasi saat plugin diaktifkan.

5. Dependency soft harus menampilkan warning dan menonaktifkan fitur terkait.

6. Dashboard harus tetap tampil walaupun sebagian widget belum aktif.

## 9\. Service Contract Antar-Plugin

## 9.1 Prinsip Service Contract

Service contract adalah aturan resmi komunikasi antar-plugin.

Setiap service contract harus memiliki:

1. Nama contract.

2. Provider plugin.

3. Consumer plugin.

4. Method/API.

5. Input schema.

6. Output schema.

7. Permission requirement.

8. Error response.

9. Audit requirement.

10. Versioning policy.

## 9.2 Larangan

Plugin tidak boleh:

1. Membaca tabel plugin lain secara langsung.

2. Mengubah data plugin lain tanpa API/service.

3. Mengandalkan struktur internal plugin lain.

4. Menggunakan undocumented endpoint.

5. Melakukan bypass permission antar-plugin.

6. Mengabaikan ABAC saat membaca data lintas plugin.

# 15\. Permission Architecture

## 15.1 Permission Naming Convention

Gunakan format:

text id="fjec7n" \<plugin\>.\<action\>

Contoh:

\`\`\`text id=“49zv4f” agenda.read agenda.create agenda.update agenda.cancel agenda.archive

duty.read duty.create duty.update duty.submit duty.approve duty.return duty.reject duty.generate\_document duty.upload\_final duty.upload\_evidence duty.verify\_evidence duty.export

dashboard.read dashboard.leadership

mmc.read mmc.create\_draft mmc.review

archive.read archive.manage audit.read

\#\# 15.2 Permission per Plugin

| Plugin | Permission Minimal |  
|---|---|  
| Agenda | \`agenda.read\`, \`agenda.create\`, \`agenda.update\`, \`agenda.cancel\`, \`agenda.archive\` |  
| ST/SPPD | \`duty.read\`, \`duty.create\`, \`duty.submit\`, \`duty.approve\`, \`duty.generate\_document\`, \`duty.upload\_evidence\`, \`duty.verify\_evidence\` |  
| Dashboard | \`dashboard.read\`, \`dashboard.leadership\` |  
| MMC | \`mmc.read\`, \`mmc.create\_draft\`, \`mmc.review\` |  
| Archive | \`archive.read\`, \`archive.manage\` |  
| Audit | \`audit.read\` |

\#\# 15.3 ABAC Rules

ABAC harus mempertimbangkan:

1\. Unit kerja user.  
2\. Faskes user.  
3\. Role user.  
4\. Status pengajuan.  
5\. Ownership data.  
6\. Klasifikasi dokumen.  
7\. Tahap approval.  
8\. Hubungan user dengan pengajuan.  
9\. Jenis dokumen.  
10\. Status arsip.

\#\# 15.4 Contoh ABAC

| Skenario | Rule |  
|---|---|  
| Pegawai melihat ST miliknya | Allowed jika \`created\_by\` atau termasuk peserta. |  
| Kepala Faskes melihat ST faskes | Allowed jika \`health\_facility\_id\` sama. |  
| Keuangan melihat SPPD | Allowed jika pengajuan berbiaya dan role keuangan. |  
| Reviewer MMC melihat draft | Allowed jika data ditandai layak publikasi. |  
| Auditor melihat dokumen | Allowed read-only sesuai klasifikasi. |  
| User biasa melihat arsip final | Denied kecuali terkait langsung. |

# 17\. Database Boundary

## 17.1 Prinsip

1. Setiap plugin memiliki tabel sendiri.

2. Nama tabel harus jelas dan konsisten.

3. Relasi lintas-plugin menggunakan reference ID dan snapshot.

4. Jangan membuat foreign key keras lintas-plugin bila menghambat modularitas, kecuali pola repo mendukung.

5. Data penting yang berasal dari plugin lain harus disimpan sebagai snapshot untuk menjaga historis.

6. Dashboard membaca melalui aggregate service/API.

## 17.2 Prefix Tabel

| Plugin | Prefix |
| :---- | :---- |
| Platform | satusehat\_ |
| Agenda | agenda\_ |
| ST/SPPD | duty\_ |
| MMC | mmc\_ atau duty\_publication\_ sesuai keputusan |
| Document Template | document\_template\_ atau duty\_template\_ sesuai pola |
| Document Archive | document\_archive\_ atau document\_archives |

## 17.3 Snapshot Rule

Snapshot wajib digunakan pada:

1. Agenda yang menjadi dasar ST.

2. Data pejabat penandatangan pada dokumen final.

3. Data pelaksana pada ST final.

4. Template version pada dokumen final.

5. Biaya/SPPD saat pengajuan disetujui.

6. Klasifikasi SPM saat tugas diterbitkan.

Tujuannya agar dokumen final tidak berubah walaupun data master berubah.

# 20\. PDF dan Document Architecture

## 20.1 Prinsip Dokumen

1. Dokumen draft boleh berubah.

2. Dokumen final tidak boleh berubah langsung.

3. Revisi dokumen final harus melalui revisi/addendum.

4. Dokumen final memiliki hash.

5. Dokumen final memiliki metadata.

6. Dokumen final masuk arsip.

7. Download dokumen final dicatat.

8. Upload dokumen final divalidasi.

# 21\. Dashboard Architecture

## 21.1 Prinsip Dashboard

1. Dashboard tidak query langsung semua tabel plugin tanpa kontrak.

2. Dashboard membaca aggregate endpoint/service.

3. Dashboard harus tetap tampil walaupun sebagian plugin nonaktif.

4. Dashboard menerapkan RBAC/ABAC.

2\. Setiap plugin terdaftar di plugin registry.

5\. Setiap plugin memiliki migration bila memerlukan tabel.  
6\. Setiap plugin memiliki permission.  
7\. Setiap plugin memiliki audit event.

9\. Dashboard membaca data melalui aggregate/service.  
10\. Plugin tidak mengubah core tanpa alasan kuat.  
11\. Plugin tetap berjalan bila dependency soft nonaktif.  
12\. Data lintas-plugin disimpan melalui reference ID dan snapshot.  
13\. Dokumen final memiliki metadata dan hash.  
14\. Akses dokumen mengikuti RBAC/ABAC.  
15\. Test minimal tersedia atau validasi manual terdokumentasi.
