## Tata Kelola Data dan Kebijakan Retensi MVP Satu Sehat Kobar v1.4

### Agenda Dinkes, ST/SPPD, Bukti Tugas, Jurnal Pegawai, Dashboard, MMC Terbatas, dan Arsip Digital Dasar

---

## 1\. Tujuan Dokumen

Dokumen ini menjadi acuan tata kelola data dan retensi untuk MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro**.

Dokumen ini digunakan untuk memastikan data yang dikelola dalam sistem:

1. Memiliki pemilik data yang jelas.

2. Memiliki klasifikasi akses yang jelas.

3. Digunakan sesuai tujuan.

4. Tidak dibuka kepada pihak yang tidak berwenang.

5. Memiliki audit trail.

6. Memiliki metadata arsip.

7. Memiliki kebijakan retensi awal.

8. Tidak mencampur data administrasi tugas dengan data klinis pasien.

9. Dapat dipertanggungjawabkan saat UAT, pilot, audit, dan pengembangan fase berikutnya.

10. Mendukung prinsip keamanan, privasi, dan tata kelola pemerintahan digital.

---

## 2\. Prinsip Tata Kelola Data

Tata kelola data Satu Sehat Kobar mengikuti prinsip berikut:

1. **Data minimization**  
   Sistem hanya menyimpan data yang diperlukan untuk Agenda, ST/SPPD, bukti tugas, jurnal, dashboard, MMC, dan arsip.

2. **Purpose limitation**  
   Data digunakan sesuai tujuan fitur, bukan untuk tujuan lain tanpa keputusan.

3. **Access control by role and attribute**  
   Akses data dikendalikan melalui RBAC dan ABAC.

4. **Private by default**  
   Data dan dokumen tidak public secara default.

5. **Auditability**  
   Perubahan dan akses penting harus tercatat.

6. **Historical integrity**  
   Dokumen final dan snapshot penting harus menjaga riwayat.

7. **Data classification**  
   Setiap data/dokumen penting harus memiliki klasifikasi akses.

8. **No uncontrolled deletion**  
   Data final, arsip, dan audit tidak boleh dihapus sembarangan.

9. **Human review for publication**  
   Data untuk MMC/publikasi harus melalui review manusia.

10. **No clinical patient data in MVP**  
    MVP tidak menyimpan data klinis pasien dalam Agenda, ST/SPPD, MMC, atau bukti tugas.

6. Backup diklasifikasikan confidential.

7. Audit log diklasifikasikan audit.

8. Data public hanya boleh berasal dari proses review resmi.

9. Data pasien tidak boleh masuk klasifikasi MVP karena tidak boleh disimpan.

10. Perubahan klasifikasi harus diaudit.

# 9\. Data Lifecycle

## 9.1 Siklus Data Agenda

text id="lifecycle-agenda-001" draft → proposed → confirmed → need\_st → st\_created → in\_progress → completed → archived

Status khusus:

text id="lifecycle-agenda-002" cancelled rescheduled

Aturan:

1. Agenda draft dapat diedit.

2. Agenda confirmed dapat menjadi dasar ST.

3. Agenda cancelled tidak dapat menjadi dasar ST baru tanpa role khusus.

4. Agenda linked ke ST menyimpan relasi.

5. Agenda completed dapat diarsipkan.

6. Perubahan status dicatat.

---

## 9.2 Siklus Data ST/SPPD

text id="lifecycle-duty-001" draft → submitted → approval\_in\_progress → returned / rejected / final\_approved → document\_issued → in\_progress → report\_submitted → evidence\_verified → completed → archived

Aturan:

1. Draft dapat diedit.

2. Submitted masuk approval.

3. Returned dapat diperbaiki.

4. Rejected tidak lanjut kecuali dibuat ulang atau kebijakan khusus.

5. Final approved menjadi dasar generate PDF.

6. Document issued menjadi dasar pelaksanaan tugas.

7. Completed setelah bukti diverifikasi.

8. Archived setelah dokumen final dan bukti sesuai.

---

## 9.3 Siklus Dokumen

text id="lifecycle-document-001" draft → generated → ready\_to\_sign → downloaded\_for\_signing → signed\_manual / signed\_external\_tte → final\_uploaded → archived

Aturan:

1. Dokumen draft dapat digenerate ulang.

2. Dokumen ready\_to\_sign dapat diunduh.

3. Dokumen final wajib PDF.

4. Dokumen final wajib hash.

5. Dokumen final tidak boleh diganti langsung.

6. Revisi dokumen final harus versi baru.

7. Dokumen final masuk arsip.

---

## 9.4 Siklus Bukti

text id="lifecycle-evidence-001" not\_uploaded → draft → submitted → returned / verified

Aturan:

1. Bukti draft dapat diedit.

2. Bukti submitted menunggu verifikasi.

3. Bukti returned harus diperbaiki.

4. Bukti verified tidak boleh dihapus user biasa.

5. Verifikasi bukti memperbarui jurnal.

6. Bukti finance tetap finance.

---

## 9.5 Siklus MMC

text id="lifecycle-mmc-001" draft → reviewed → approved\_for\_publication → published / not\_published

Aturan:

1. Draft MMC hanya dari laporan verified.

2. Draft wajib review manusia.

3. Draft tidak boleh otomatis publish.

4. Data pasien dan rincian biaya tidak boleh masuk.

5. Link publikasi disimpan setelah publish nyata.

---

## 9.6 Siklus Arsip

text id="lifecycle-archive-001" registered → active\_archive → inactive\_archive → permanent / reassess / eligible\_for\_disposition

Aturan MVP:

1. MVP hanya mencatat metadata retensi awal.

2. Pemusnahan otomatis tidak masuk MVP.

3. Arsip final tidak boleh dihapus user biasa.

4. Perubahan status arsip harus diaudit.

5. Access log wajib untuk dokumen restricted/finance.

# 10\. Retention Policy

## 10.1 Prinsip Retensi

Kebijakan retensi awal mengikuti prinsip:

1. Retensi disesuaikan dengan jenis dokumen.

2. Retensi tidak otomatis memusnahkan data pada MVP.

3. Retensi digunakan sebagai metadata dan pengingat.

4. Pemusnahan arsip memerlukan SOP/kebijakan resmi.

5. Arsip final tidak boleh dihapus sembarangan.

6. Audit log dipertahankan sesuai kebutuhan audit.

7. Backup mengikuti kebijakan backup terpisah.

8. Data training/dummy dapat dihapus setelah pelatihan dengan persetujuan admin.

9. Data publikasi MMC mengikuti kebijakan publikasi dan arsip informasi publik.

10. Perubahan retensi dicatat dalam decision log.

---

| Kategori | Contoh | Perlakuan MVP |
| :---- | :---- | :---- |
