# API\_SERVICE\_CONTRACT.md

## Kontrak API dan Service MVP Satu Sehat Kobar v1.4

### Agenda Dinkes, ST/SPPD, SPM, MMC, Dashboard, Dokumen, Bukti, Jurnal, dan Arsip

---

## 1\. Tujuan Dokumen

Dokumen ini menjelaskan kontrak API dan service untuk MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro plugin ecosystem**.

Dokumen ini menjadi acuan untuk:

1. Mengembangkan API antar-plugin.

2. Menentukan batas akses data antar-plugin.

3. Menentukan standar request dan response.

4. Menentukan permission pada setiap endpoint.

5. Menentukan validasi input.

6. Menentukan error response.

7. Menentukan audit event.

8. Menentukan service contract lintas-plugin.

9. Mencegah plugin membaca tabel internal plugin lain secara langsung.

10. Menjaga implementasi tetap sesuai scope MVP.

---

## 2\. Prinsip API dan Service Contract

## 2.1 Prinsip Utama

1. Semua endpoint state-changing wajib authentication.

2. Semua endpoint wajib permission check.

3. Semua endpoint yang mengakses data lintas unit/faskes wajib ABAC.

4. API tidak boleh membuka data di luar kewenangan user.

5. API antar-plugin harus menggunakan service contract.

6. Plugin tidak boleh membaca tabel internal plugin lain secara langsung.

7. Semua input harus divalidasi.

8. Semua output harus disaring sesuai role dan klasifikasi data.

9. Semua aksi penting wajib masuk audit trail.

10. Response error harus konsisten.

11. Endpoint harus mendukung pagination untuk daftar data.

12. Endpoint export harus dibatasi role.

13. Endpoint dokumen dan arsip harus mencatat access log.

14. Integrasi eksternal penuh tidak masuk MVP.

15. API harus dapat berjalan tanpa TTE/BSrE, SRIKANDI, SIMPEG, SIPD, e-Kinerja, dan SATUSEHAT Kemenkes.

## 5\. Standar Authentication, RBAC, dan ABAC

## 5.1 Authentication

Semua endpoint internal MVP wajib memeriksa user login, kecuali endpoint health check bila ada.

Response bila belum login:

json id="blnh3p" {   "error": {     "code": "UNAUTHENTICATED",     "message": "Anda harus login untuk mengakses resource ini.",     "details": {}   } }

## 5.2 RBAC

Setiap endpoint wajib mendefinisikan permission minimal.

Contoh:

text id="n4aw2b" GET /api/agenda/events          requires agenda.read POST /api/agenda/events         requires agenda.create POST /api/duty-requests         requires duty.create POST /api/duty-requests/:id/approve requires duty.approve POST /api/duty-documents/:id/upload-signed requires duty.upload\_final GET /api/satusehat/dashboard    requires dashboard.read

## 5.3 ABAC

ABAC wajib diterapkan pada:

1. Data per unit kerja.

2. Data per faskes.

3. Data milik pegawai.

4. Dokumen restricted.

5. Dokumen finance.

6. Bukti tugas.

7. Approval sesuai tahap.

8. Jurnal pegawai.

9. Arsip final.

10. Draft MMC.

Contoh ABAC:

| Skenario | Rule |
| :---- | :---- |
| Pegawai membaca ST miliknya | Allowed jika user adalah pemohon atau peserta tugas. |
| Kepala Faskes membaca ST faskes | Allowed jika health\_facility\_id sama. |
| Keuangan membaca SPPD | Allowed jika role keuangan dan pengajuan berbiaya. |
| Reviewer MMC membaca draft | Allowed jika user punya mmc.review. |
| Auditor membaca arsip | Read-only sesuai klasifikasi. |
| Pegawai membaca arsip keuangan | Denied kecuali termasuk pihak terkait dan memiliki permission. |

# 11\. Approval API Contract

## 11.1 Approve Duty Request

### Endpoint

text id="yq2u1f" POST /api/duty-requests/:id/approve

### Permission

text id="z67v6c" duty.approve

### Request

json id="0ual4s" {   "approval\_step": "supervisor",   "note": "Disetujui." }

### Business Rules

1. User harus menjadi approver pada tahap aktif.

2. Role yang bukan tahapnya tidak dapat approve.

3. Setelah approve, sistem mengaktifkan tahap berikutnya.
