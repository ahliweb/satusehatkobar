# DATABASE\_MVP\_SCHEMA.md

## Rancangan Database MVP Satu Sehat Kobar v1.4

### Agenda Dinkes, ST/SPPD, Bukti Tugas, SPM, Jurnal Pegawai, Dashboard, MMC Terbatas, dan Arsip Dasar

---

## 1\. Tujuan Dokumen

Dokumen ini menjelaskan rancangan database MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro plugin ecosystem**.

Dokumen ini menjadi acuan untuk:

1. Membuat migration database.

2. Membuat seed data awal.

3. Menentukan relasi antar-entitas.

4. Menentukan batas tabel antar-plugin.

5. Menentukan field minimal untuk MVP.

6. Menentukan indeks awal.

7. Menentukan status lifecycle.

8. Menentukan audit trail.

9. Menentukan metadata dokumen dan arsip.

10. Mencegah AI coding agent membuat struktur database yang melebar dari scope MVP.

---

## 2\. Prinsip Desain Database

## 2.1 Prinsip Utama

1. Database mengikuti pola yang sudah ada di repository AWCMS-Micro.

2. Setiap plugin memiliki tabel sendiri.

3. Nama tabel harus konsisten, eksplisit, dan mudah dipahami.

4. Data lintas-plugin menggunakan reference ID dan snapshot.

5. Plugin tidak boleh bergantung langsung pada struktur internal plugin lain tanpa service contract.

6. Dokumen final harus memiliki metadata, hash, status, dan versi.

7. Semua aksi penting harus tercatat di audit trail.

8. Semua tabel utama memiliki status lifecycle.

9. Semua tabel utama memiliki created\_at dan updated\_at.

10. Semua tabel utama yang dibuat user memiliki created\_by dan bila relevan updated\_by.

11. Data pribadi dibatasi sesuai kebutuhan operasional.

12. Data pasien klinis tidak disimpan dalam plugin Agenda atau ST/SPPD.

13. Migration harus aman, bertahap, dan tidak destruktif.

14. Field retensi dan arsip disiapkan sejak MVP.

15. Index dibuat untuk query utama dashboard, filter, dan workflow.

## 2.2 Prinsip Modularitas

1. Tabel agenda\_\* hanya dimiliki oleh plugin Agenda.

2. Tabel duty\_\* hanya dimiliki oleh plugin ST/SPPD.

3. Tabel document\_\* dimiliki oleh plugin Document Archive/Template.

4. Tabel satusehat\_\* dimiliki oleh platform Satu Sehat Kobar.

5. Tabel mmc\_\* atau duty\_publication\_\* dimiliki oleh plugin MMC/atau bagian duty publication bila MVP menyatukan sementara.

6. Dashboard tidak mengubah tabel plugin; dashboard hanya membaca aggregate data.

7. Arsip menyimpan metadata final, bukan mengganti dokumen sumber.

## 3.3 Tipe Data

Tipe data aktual mengikuti database yang dipakai repository. Bila menggunakan SQLite/D1-like SQL, tipe dapat disesuaikan menjadi TEXT, INTEGER, REAL, dan DATETIME.

Panduan umum:

| Kebutuhan | Tipe Umum |
| :---- | :---- |
| ID | TEXT / UUID string |
| Nama/judul | TEXT |
| Deskripsi panjang | TEXT |
| Status | TEXT |
| Boolean | INTEGER 0/1 |
| Tanggal/waktu | TEXT ISO datetime atau DATETIME sesuai pola |
| Angka biaya | INTEGER atau REAL sesuai standar repo |
| JSON | TEXT berisi JSON bila DB tidak punya tipe JSON native |

# 17\. Data Classification

## 17.1 Klasifikasi Data

| Kategori | Contoh |
| :---- | :---- |
| Public | Berita MMC yang sudah disetujui. |
| Internal | Agenda internal, ST umum. |

| Confidential | Dokumen kepegawaian sensitif, catatan tertentu. |

1. Public dapat muncul di publikasi setelah review.

2. Internal hanya untuk user login sesuai unit/role.

3. Restricted hanya untuk user terkait dan pejabat berwenang.

4. Confidential hanya untuk role tertentu.

5. Finance hanya untuk keuangan, pimpinan, auditor, dan pihak terkait sesuai kewenangan.

# 19\. Data Integrity Rules

## 19.1 Agenda

1. end\_at tidak boleh sebelum start\_at.

2. Agenda public tidak otomatis masuk MMC.

3. Agenda cancelled tidak boleh menjadi dasar ST baru kecuali role khusus.

4. Agenda yang sudah linked ke ST tidak boleh dihapus permanen pada MVP.

## 19.2 Duty Request

1. Pengajuan tidak dapat submit jika field wajib kosong.

2. Pengajuan ST dari agenda wajib menyimpan agenda\_snapshot.

3. Pengajuan berbiaya wajib memiliki minimal satu budget line.

4. Pengajuan tidak dapat menjadi final\_approved tanpa approval sesuai chain.

5. Pengajuan tidak dapat completed tanpa bukti verified.

6. Pengajuan final tidak boleh berubah langsung; harus revisi/addendum.

## 19.3 Document

1. PDF final wajib PDF.

2. Dokumen final wajib memiliki hash.

3. Dokumen final tidak dapat diganti langsung.

4. Download dokumen final harus tercatat bila dokumen restricted/finance/confidential.

5. Template version wajib disimpan saat dokumen dibuat.

## 19.4 Evidence

1. File bukti harus melewati validasi upload.

2. Bukti finance hanya dapat dilihat role berwenang.

3. Return bukti wajib catatan.

4. Verify bukti wajib reviewer.

## 19.5 Archive

1. Arsip final wajib memiliki file path.

2. Arsip final wajib memiliki hash.

3. Arsip final wajib memiliki klasifikasi akses.

4. Akses arsip restricted/finance/confidential wajib dicatat.

5. Retensi tidak boleh kosong untuk dokumen final bila rule tersedia.

# 20\. Soft Delete and Hard Delete

## 20.1 Soft Delete

Soft delete dapat digunakan untuk:

1. Agenda draft.

2. Lampiran agenda.

3. Draft pengajuan.

4. Bukti yang salah upload sebelum submit.

5. Draft MMC.

## 20.2 Hard Delete

Hard delete tidak direkomendasikan pada MVP untuk:

1. ST final.

2. SPPD final.

3. Dokumen final.

4. Bukti verified.

5. Audit log.

6. Arsip final.

7. Data approval.

## 20.3 Aturan

1. Hard delete hanya role khusus.

2. Hard delete harus audit.

3. Hard delete tidak boleh dilakukan pada dokumen final kecuali sesuai kebijakan arsip.

4. Prefer cancelled, archived, atau inactive.
