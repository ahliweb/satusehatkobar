## 1. Tujuan

Dokumen ini memverifikasi paket `docs/prd` dan menambahkan referensi teknis implementasi agar developer, reviewer, dan AI coding agent tidak mengeksekusi PRD secara terisolasi.

Dokumen ini tidak mengganti PRD, arsitektur plugin, skema database, atau kontrak API. Dokumen ini menjadi peta rujukan dari requirement ke permukaan implementasi yang benar — termasuk **bagaimana** requirement PRD diterjemahkan ke runtime AWCMS-Micro/EmDash yang sebenarnya, bukan hanya **di mana** file ditempatkan.

## 4. Runtime AWCMS-Micro + EmDash: Format Plugin, Penamaan, dan Persistensi

Bagian ini adalah hasil verifikasi terhadap workspace nyata (`awcmsmicro-dev/`, plugin contoh `awcms-micro-sikesra`, dan template `awcms-sskobar-cloudflare`). Tujuannya memastikan PRD dapat diimplementasikan **mengikuti grain framework**, bukan melawannya.

### 4.1 Format Plugin: Native vs Standard

EmDash punya dua format plugin:

| Format | Cara Daftar | Admin UI | Akses Data | Bisa Sandbox/Marketplace? |
|--------|-------------|----------|------------|---------------------------|
| **Standard** | `definePlugin({ hooks, routes })` | Block Kit | via `ctx` | Ya |
| **Native** | descriptor dengan `format: "native"`, `id`, `version`, `adminEntry` | React (`admin.tsx`) | akses D1 penuh via `ctx.db` (Kysely) — default SSK; juga `ctx.kv`, `ctx.storage` | Tidak — hanya jalan di host isolate (`plugins: []`) |

**Keputusan untuk SSK:** Seluruh plugin MVP Satu Sehat Kobar adalah **plugin Native**, sama seperti `awcms-micro-sikesra`. Alasannya: PRD mensyaratkan admin UI React (dashboard, form ST/SPPD, approval, arsip) dan logika domain yang butuh akses data langsung. Konsekuensi:

- Plugin SSK didaftarkan di array `plugins: []` template (bukan `sandboxed: []`).
- Plugin SSK **tidak** dipublish ke marketplace dan tidak berjalan dalam mode sandbox.
- Setiap plugin terdiri dari **descriptor** (`src/index.ts`, dieksekusi Vite saat build, harus side-effect-free) dan **definition** runtime (`definePlugin()` di entry terpisah, dieksekusi saat request). Admin React diekspos via `adminEntry`/`admin.tsx`.

Referensi format dan anatomi plugin: `awcmsmicro-dev/skills/creating-plugins/SKILL.md`. Contoh lengkap: `awcmsmicro-dev/packages/plugins/awcms-micro-sikesra/`.

### 4.2 Konvensi Penamaan Paket Plugin

PRD doc 03 memakai **plugin key logis** (mis. `duty-travel`). Di workspace, key logis tersebut diwujudkan dengan konvensi konkret berikut:

| Elemen | Pola | Contoh (existing) | Contoh (SSK baru) |
|--------|------|-------------------|-------------------|
| Direktori paket | `awcmsmicro-dev/packages/plugins/awcms-micro-<key>/` | `awcms-micro-sikesra/` | `awcms-micro-duty-travel/` |
| Nama npm | `@awcms-micro/plugin-<key>` | `@awcms-micro/plugin-sikesra` | `@awcms-micro/plugin-duty-travel` |
| Slug manifest (`emdash-plugin.jsonc`) | `awcms-micro-<key>` | `awcms-micro-sikesra` | `awcms-micro-duty-travel` |
| Descriptor `id` | sesuai konstanta plugin | `AWCMS_SIKESRA_PLUGIN_ID` | konstanta plugin masing-masing |

Plugin key logis di PRD (`satusehat-dashboard`, `agenda-dinkes`, `duty-travel`, `document-template`, `document-archive`, `spm-health`, `mmc-publication`, `official-region`, `local-region`, `health-facility`, `personnel`) dipetakan ke nama paket `awcms-micro-<key>`. Doc 03 sudah mencantumkan kolom nama paket `awcms-micro-*`; **gunakan kolom itu sebagai nama direktori dan slug final.**

### 4.3 Registrasi Plugin di Template

Plugin baru tidak aktif sampai didaftarkan di template Cloudflare SSK:

1. Tambahkan dependency paket plugin di `awcmsmicro-dev/templates/awcms-sskobar-cloudflare/package.json`.
2. Import factory descriptor dan masukkan ke array `plugins: []` di `awcmsmicro-dev/templates/awcms-sskobar-cloudflare/astro.config.mjs` (pola sama dengan `awcmsMicroExamplePlugin(...)` dan `awcmsMicroGalleryPlugin(...)`).
3. Jika plugin harus survive rebuild dari upstream, tambahkan protected path (lihat §5).
4. Jalankan generate types dan validasi sesuai `emdash-cli` skill dan `docs/deployment/cloudflare.md`.

### 4.4 Model Persistensi — PENTING (default: D1 langsung via `ctx.db`)

> **Default ditetapkan oleh DEC-019:** seluruh plugin SSK mempersistensi data domain **langsung di D1** melalui `ctx.db` (query builder Kysely di atas binding `DB`). Skema relasional di `04.DATABASE_MVP_SCHEMA.docx.md` adalah **skema fisik** yang diimplementasikan apa adanya — bukan sekadar skema logis. Lihat `12.CHANGE_CONTROL_AND_DECISION_LOG.docx.md` DEC-019.

Plugin EmDash Native menerima `ctx.db` bertipe `Kysely<Database>` dengan **akses penuh** ke database D1. Mekanisme yang tersedia dan kapan dipakai:

| Mekanisme | API | Status untuk SSK | Catatan |
|-----------|-----|------------------|---------|
| **D1 langsung** | `ctx.db` (Kysely) di atas binding `DB` | **DEFAULT** untuk semua data domain | Tabel, kolom, relasi, dan index persis Doc 04. Query via Kysely; relasi lintas plugin di-enforce di service layer (bukan FK runtime rapuh). |
| **KV** | `ctx.kv.get/set/delete` (mendukung TTL) | Dipakai untuk **cache** saja | Cache dashboard 15 menit (DEC-014). Bukan store domain. |
| **EmDash content collections** | schema-in-DB via `seed/seed.json` | Untuk entitas mirip konten yang dikelola lewat admin EmDash | Mis. konten publik/site; bukan untuk data transaksional ST/SPPD. |
| **Plugin storage collections** | `ctx.storage.<collection>` (deklaratif + index) | **Pengecualian**, harus dicatat sebagai keputusan tersendiri | Pola yang dipakai `awcms-micro-sikesra`; tidak dipakai sebagai store domain default SSK. |

**Migrasi skema milik plugin.** EmDash punya migration runner internal untuk tabel core-nya sendiri, dan peta migrasi itu **frozen** — plugin **tidak bisa** menyuntik migrasi ke runner core. Karena itu, dengan D1 langsung **setiap plugin mengelola lifecycle skema-nya sendiri**:

- Jalankan DDL **idempoten** (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, dan alter terurut) via `ctx.db` pada lifecycle hook `install`/`activate`.
- Lacak versi skema dengan tabel migrasi milik plugin, mis. `<prefix>_migrations` (`duty_migrations`, `agenda_migrations`, …).
- Migrasi harus aman dijalankan ulang (first request setelah deploy bisa memicu eksekusi).

**Implikasi implementasi untuk SSK:**

- Implementasi fisik default = **D1 langsung (`ctx.db`)** mengikuti Doc 04; `ctx.kv` hanya untuk cache.
- "KV cache 15 menit" untuk dashboard = `ctx.kv` dengan TTL, **bukan** binding KV terpisah. Template hanya membind KV `SESSION` (lihat §4.5).
- Prefix tabel tetap mengikuti pemilik plugin (`duty_*`, `agenda_*`, dst.) agar boundary kepemilikan data jelas; plugin tidak membaca tabel milik plugin lain langsung — gunakan service contract/snapshot.
- Memakai storage collections atau `ctx.kv` sebagai store domain untuk kasus khusus adalah **pengecualian** yang harus dicatat sebagai keputusan tersendiri di Doc 12.

### 4.5 Binding Cloudflare Template SSK

Template `awcms-sskobar-cloudflare/wrangler.jsonc` mengekspos binding berikut. Pakai nama binding ini, jangan mengarang nama baru:

| Binding | Jenis | Peran | Catatan PRD |
|---------|-------|-------|-------------|
| `DB` | D1 | Database utama EmDash + data plugin | EmDash `database: d1({ binding: "DB" })` |
| `MEDIA` | R2 | Storage file: media, upload, evidence, arsip | EmDash `storage: r2({ binding: "MEDIA" })`; akses via signed URL/proxy |
| `SESSION` | KV | Session store | Satu-satunya KV namespace; cache plugin pakai `ctx.kv` |
| `IMAGES` | Images | Transformasi gambar | Untuk media/gallery |
| `LOADER` | Worker Loader | Runner sandbox plugin | Disediakan walau plugin SSK Native |

Jika sebuah fitur PRD membutuhkan binding baru (mis. KV cache terpisah, queue, atau Durable Object), itu adalah perubahan infrastruktur yang harus melalui change control dan update `wrangler.jsonc` + `docs/deployment/cloudflare.md`.

## 7. Referensi Teknis per Aktivitas

### 7.1 Membuat Plugin Baru

1. Baca **§4** dokumen ini dan `awcmsmicro-dev/skills/creating-plugins/SKILL.md` untuk format plugin (Native untuk SSK), penamaan paket, descriptor/definition split, dan registrasi.
2. Baca `docs/prd/03.PLUGIN_ARCHITECTURE.docx.md` untuk boundary, service contract, dan struktur plugin.
3. Baca `docs/awcms-micro-implementation-boundaries.md` untuk memastikan lokasi plugin rebuild-safe.
4. Buat plugin di `awcmsmicro-dev/packages/plugins/awcms-micro-<key>/` hanya jika tidak cukup sebagai fitur dalam plugin existing; ikuti pola `awcms-micro-sikesra` (descriptor `src/index.ts` + definition + `admin.tsx` + `emdash-plugin.jsonc`).
5. Tambahkan dokumentasi plugin-local: `README.md`, `docs/api.md`, `docs/permissions.md`, dan tests. Persistensi default = D1 langsung dengan modul migrasi idempoten milik plugin (lihat §7.2).
6. Daftarkan plugin di `awcms-sskobar-cloudflare/astro.config.mjs` (`plugins: []`) dan `package.json` template (lihat §4.3).
7. Pastikan admin sidebar plugin muncul dalam group sendiri sesuai AGENTS root.

### 7.2 Mempersistensi Data Plugin

> Baca §4.4 lebih dulu. **Default SSK = D1 langsung via `ctx.db`** (DEC-019); Doc 04 adalah skema fisik. `ctx.kv` hanya untuk cache.

1. Baca `docs/prd/04.DATABASE_MVP_SCHEMA.docx.md` dan addendum master data terkait sebagai **skema fisik** (tabel, kolom, relasi, index, prefix, kepemilikan).
2. Implementasikan tabel di D1 via `ctx.db` (Kysely). Tulis modul migrasi **idempoten** milik plugin (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, alter terurut) yang dijalankan pada lifecycle hook `install`/`activate`, dengan tabel versi `<prefix>_migrations` (mis. `duty_migrations`).
3. Pakai prefix nama tabel sesuai plugin owner (`duty_*`, `agenda_*`, dst.). Plugin tidak membaca tabel milik plugin lain langsung — gunakan service contract/snapshot.
4. Jangan membuat FK lintas plugin sebagai dependency runtime yang rapuh; enforce relasi di service layer dan snapshot data historis.
5. Semua query memakai parameter binding (jangan string-concat). Validasi mengikuti `docs/deployment/d1-database.md` dan `docs/awcms-micro-d1-mirror-sync.md` jika memakai mirror lokal.
6. Gunakan `ctx.kv` (dengan TTL) hanya untuk cache, mis. metrics dashboard 15 menit. Jangan asumsikan binding KV terpisah — template hanya membind `SESSION` (§4.5).
7. Memakai EmDash storage collections (`ctx.storage`) atau `ctx.kv` sebagai store domain adalah **pengecualian** yang harus dicatat sebagai keputusan tersendiri di Doc 12.

### 7.3 Membuat Endpoint API

1. Baca `docs/prd/05.API Service Contract Satu Sehat Kobar v1.4.docx.md`.
2. Handler wajib mengikuti urutan: auth middleware, RBAC, ABAC, validation, service call, audit, response envelope.
3. Gunakan response envelope standar `{ success, data, meta }` atau `{ success, error }`.
4. Error message untuk user harus dalam Bahasa Indonesia.
5. Endpoint yang menyentuh file harus mengikuti `docs/deployment/r2-storage.md` dan security upload checklist.

### 7.4 Mengimplementasikan RBAC/ABAC

1. Baca `docs/prd/10.Security and Privacy Checklist.docx.md` dan `docs/security/access-control.md`.
2. RBAC menentukan boleh tidaknya action dilakukan.
3. ABAC menentukan scope data: unit, faskes, pemilik, bawahan, atau keuangan scope.
4. Jangan hanya menyembunyikan tombol di UI; enforcement wajib ada di service layer.
5. Access denied harus dicatat sebagai audit/security event jika relevan.

### 7.5 Upload Dokumen, Evidence, dan Arsip

1. Baca Doc 05 untuk endpoint upload dan Doc 10 untuk security checklist.
2. Simpan file di R2; database hanya menyimpan metadata, hash, classification, owner, dan status.
3. Jangan expose R2 key mentah ke client; gunakan signed URL atau endpoint proxy yang diaudit.
4. Arsip final harus immutable sesuai `document-archive` contract.

### 7.6 Testing dan UAT

1. Unit test untuk service dan repository per plugin.
2. Integration test untuk API contract, RBAC/ABAC, audit log, upload, dan state transition.
3. E2E/UAT mengikuti `docs/prd/06.UAT and Deployment Checklist.docx.md`.
4. Smoke deployment mengikuti `docs/deployment/cloudflare.md`.

## 8. Gap yang Harus Dikunci Sebelum Coding

Sebelum membuat issue implementasi dari backlog, kunci hal berikut:

| Gap | Risiko | Tindakan Wajib |
|-----|--------|----------------|
| **Migrasi skema milik plugin**: default D1 langsung (DEC-019) tetapi EmDash tak punya plugin migration runner | Tabel tidak terbuat saat deploy, atau migrasi non-idempoten gagal di first request | Setiap plugin punya modul migrasi idempoten (`<prefix>_migrations`) yang jalan di `install`/`activate` via `ctx.db`. ## 9. Checklist Siap Implementasi

Sebuah issue PRD siap dikerjakan jika semua item berikut terpenuhi:

- [ ] Dokumen PRD sumber sudah disebutkan di issue.
- [ ] Plugin owner, nama paket (`awcms-micro-<key>`), dan prefix tabel sudah jelas.
- [ ] Format plugin (Native) dan titik registrasi (`astro.config.mjs` `plugins: []`) sudah jelas.
- [ ] Lokasi implementasi di `awcmsmicro-dev/` sudah sesuai protected boundary.
- [ ] API contract atau service contract sudah jelas.
- [ ] RBAC permission dan ABAC scope sudah jelas.
- [ ] Audit event untuk mutasi sudah didefinisikan.
- [ ] Persistensi: tabel D1 sesuai Doc 04 + modul migrasi idempoten plugin sudah direncanakan (§4.4, §7.2); pengecualian non-D1 sudah dicatat di Doc 12 bila ada.
- [ ] Upload/R2 impact sudah diketahui jika ada file (binding `MEDIA`).
- [ ] Test path sudah jelas: unit, integration, E2E/UAT, atau smoke.
- [ ] Jika ada perubahan scope/schema/arsitektur/binding, change control sudah dibuat.
