# Technical Implementation References — Satu Sehat Kobar

**Versi:** 1.0  
**Tanggal:** Juni 2026  
**Status:** Active  
**Fungsi:** Jembatan antara PRD Satu Sehat Kobar dan referensi teknis implementasi di workspace AWCMS-Micro.

---

## 1. Tujuan

Dokumen ini memverifikasi paket `docs/prd` dan menambahkan referensi teknis implementasi agar developer, reviewer, dan AI coding agent tidak mengeksekusi PRD secara terisolasi.

Dokumen ini tidak mengganti PRD, arsitektur plugin, skema database, atau kontrak API. Dokumen ini menjadi peta rujukan dari requirement ke permukaan implementasi yang benar.

- PRD utama menetapkan 7 plugin MVP, sementara addendum 21 menambahkan 3 plugin pendukung master data. Ketiga plugin pendukung harus diperlakukan sebagai dependency implementasi, bukan pengganti 7 plugin MVP.
- Backlog masih memuat beberapa label plugin generik seperti `evidence`, `journal`, dan `approval-workflow`; secara implementasi MVP label tersebut harus dipetakan ke plugin pemilik yang benar, terutama `duty-travel`.
- Beberapa endpoint di backlog memakai path singkat seperti `/agenda`; implementasi harus mengikuti namespace final di kontrak API dan plugin router.

## 6. Referensi Teknis per Aktivitas

### 6.1 Membuat Plugin Baru

1. Baca `docs/prd/03.PLUGIN_ARCHITECTURE.docx.md` untuk boundary, service contract, dan struktur plugin.
2. Baca `docs/awcms-micro-implementation-boundaries.md` untuk memastikan lokasi plugin rebuild-safe.
3. Buat plugin di `awcmsmicro-dev/packages/plugins/` hanya jika tidak cukup sebagai fitur dalam plugin existing.
4. Tambahkan dokumentasi plugin-local: `README.md`, `docs/api.md`, `docs/permissions.md`, migrations, seeds, dan tests.
5. Pastikan admin sidebar plugin muncul dalam group sendiri sesuai AGENTS root.

### 6.2 Membuat Migration dan Seed D1

1. Baca `docs/prd/04.DATABASE_MVP_SCHEMA.docx.md` dan addendum master data terkait.
2. Gunakan prefix tabel sesuai plugin owner.
3. Jangan membuat FK lintas plugin sebagai dependency runtime yang rapuh; enforce relasi di service layer dan snapshot data historis.
4. Semua query memakai parameter binding D1.
5. Jalankan validasi D1 sesuai `docs/deployment/d1-database.md` dan `docs/awcms-micro-d1-mirror-sync.md` jika memakai mirror lokal.

### 6.3 Membuat Endpoint API

1. Baca `docs/prd/05.API Service Contract Satu Sehat Kobar v1.4.docx.md`.
2. Handler wajib mengikuti urutan: auth middleware, RBAC, ABAC, validation, service call, audit, response envelope.
3. Gunakan response envelope standar `{ success, data, meta }` atau `{ success, error }`.
4. Error message untuk user harus dalam Bahasa Indonesia.
5. Endpoint yang menyentuh file harus mengikuti `docs/deployment/r2-storage.md` dan security upload checklist.

### 6.4 Mengimplementasikan RBAC/ABAC

1. Baca `docs/prd/10.Security and Privacy Checklist.docx.md` dan `docs/security/access-control.md`.
2. RBAC menentukan boleh tidaknya action dilakukan.
3. ABAC menentukan scope data: unit, faskes, pemilik, bawahan, atau keuangan scope.
4. Jangan hanya menyembunyikan tombol di UI; enforcement wajib ada di service layer.
5. Access denied harus dicatat sebagai audit/security event jika relevan.

### 6.5 Upload Dokumen, Evidence, dan Arsip

1. Baca Doc 05 untuk endpoint upload dan Doc 10 untuk security checklist.
2. Simpan file di R2; database hanya menyimpan metadata, hash, classification, owner, dan status.
3. Jangan expose R2 key mentah ke client; gunakan signed URL atau endpoint proxy yang diaudit.
4. Arsip final harus immutable sesuai `document-archive` contract.

### 6.6 Testing dan UAT

1. Unit test untuk service dan repository per plugin.
2. Integration test untuk API contract, RBAC/ABAC, audit log, upload, dan state transition.
3. E2E/UAT mengikuti `docs/prd/06.UAT and Deployment Checklist.docx.md`.
4. Smoke deployment mengikuti `docs/deployment/cloudflare.md`.

## 7. Gap yang Harus Dikunci Sebelum Coding

Sebelum membuat issue implementasi dari backlog, kunci hal berikut:

| Gap | Risiko | Tindakan Wajib |
|-----|--------|----------------|
| Nama tabel agenda di backlog (`satusehat_agenda`) berbeda dari schema (`agenda_events`) | Migration salah dan API tidak sinkron | Gunakan Doc 04 sebagai sumber schema; update issue/backlog saat dibuat. |
| Label plugin `approval-workflow`, `evidence`, `journal` tidak termasuk 7 plugin MVP | Folder/plugin duplikat | Implementasikan sebagai domain internal `duty-travel` kecuali ada keputusan change control baru. |
| Addendum master data menambah 3 plugin pendukung | Dependency terlambat diketahui | Masukkan `official-region` dan `health-facility` sebagai prerequisite Sprint 1; `local-region` sebagai Should Have. |
| Endpoint backlog tidak selalu memakai namespace final | Route mismatch antara frontend dan backend | Ikuti Doc 05 sebagai kontrak API final. |
| Skema pegawai belum masuk Doc 04 utama secara penuh | Snapshot ST/SPPD tidak lengkap | Gunakan Doc 23 untuk employee/profile/snapshot sampai Doc 04 direvisi. |

Perubahan yang mengubah scope, plugin owner, atau schema harus dicatat di `12.CHANGE_CONTROL_AND_DECISION_LOG.docx.md`.

## 8. Checklist Siap Implementasi

Sebuah issue PRD siap dikerjakan jika semua item berikut terpenuhi:

- [ ] Dokumen PRD sumber sudah disebutkan di issue.
- [ ] Plugin owner dan prefix tabel sudah jelas.
- [ ] Lokasi implementasi di `awcmsmicro-dev/` sudah sesuai protected boundary.
- [ ] API contract atau service contract sudah jelas.
- [ ] RBAC permission dan ABAC scope sudah jelas.
- [ ] Audit event untuk mutasi sudah didefinisikan.
- [ ] Migration/seed impact sudah diketahui.
- [ ] Upload/R2 impact sudah diketahui jika ada file.
- [ ] Test path sudah jelas: unit, integration, E2E/UAT, atau smoke.
- [ ] Jika ada perubahan scope/schema/arsitektur, change control sudah dibuat.

## 9. Urutan Baca Teknis Minimum

Untuk pekerjaan coding, urutan minimum adalah:

1. `20.Master Document Index and Implementation Guide.docx.md`
2. Dokumen PRD/domain yang relevan (`PRD`, Doc 02–05, atau addendum 21–23)
3. Dokumen ini (`24.TECHNICAL_IMPLEMENTATION_REFERENCES.md`)
4. `docs/repository-structure.md`
5. `docs/awcms-micro-implementation-boundaries.md`
6. `docs/implementation-instructions.md`
7. Dokumen deployment/security yang sesuai dengan area kerja

Jika implementasi menyentuh Cloudflare bindings, D1, R2, KV, deployment, atau secret, baca `docs/deployment/` dan `docs/security/` sebelum coding.
