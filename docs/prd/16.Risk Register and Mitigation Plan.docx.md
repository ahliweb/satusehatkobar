---

## 1\. Tujuan Dokumen

Dokumen ini menjadi acuan pengelolaan risiko dalam implementasi MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro**.

Dokumen ini digunakan untuk:

1. Mengidentifikasi risiko sejak awal.

2. Menentukan tingkat dampak dan kemungkinan risiko.

3. Menetapkan mitigasi.

4. Menetapkan pemilik risiko.

5. Menentukan indikator peringatan dini.

6. Menentukan mekanisme eskalasi.

7. Menjaga scope MVP tetap terkendali.

8. Menjaga keamanan dan privasi data.

9. Menjaga kesiapan deployment dan pilot.

10. Menjadi dasar monitoring risiko mingguan/bulanan.

11. Menjadi dasar keputusan Product Owner sebelum go-live pilot.

12. Menjadi panduan bagi AI coding agent agar berhenti bila menemukan risiko tinggi.

---

## 2\. Prinsip Pengelolaan Risiko

Pengelolaan risiko MVP mengikuti prinsip:

1. **Preventive first.**  
   Risiko dicegah sejak desain, bukan hanya diperbaiki setelah terjadi.

2. **Security by default.**  
   Akses data, dokumen, bukti, dan arsip harus ditolak jika hak akses tidak jelas.

3. **Scope control.**  
   Fitur di luar MVP tidak boleh masuk implementasi tanpa keputusan Product Owner.

4. **Data protection.**  
   Data pribadi, dokumen restricted, dokumen finance, dan bukti tugas harus dilindungi.

5. **Auditability.**  
   Aksi penting harus tercatat agar dapat ditelusuri.

6. **Rollback readiness.**  
   Deployment dan migration harus memiliki rencana rollback.

7. **Human approval.**  
   Keputusan administratif, publikasi MMC, approval, dan verifikasi bukti tetap memerlukan manusia.

8. **Pilot learning.**  
   Masa pilot digunakan untuk menemukan risiko nyata, memperbaiki SOP, dan meningkatkan sistem.

9. **No silent failure.**  
   Error penting tidak boleh disembunyikan tanpa catatan.

10. **Escalate early.**  
    Risiko tinggi harus segera dieskalasi sebelum berdampak luas.

# 3\. Metodologi Penilaian Risiko

## 3.1 Skor Kemungkinan

| Skor | Level | Definisi |
| ----: | :---- | :---- |
| 1 | Rendah | Kecil kemungkinan terjadi. |
| 2 | Sedang | Mungkin terjadi pada kondisi tertentu. |
| 3 | Tinggi | Besar kemungkinan terjadi pada implementasi/pilot. |

---

## 3.2 Skor Dampak

| Skor | Level | Definisi |
| ----: | :---- | :---- |
| 1 | Rendah | Dampak kecil, tidak menghambat fitur utama. |
| 2 | Sedang | Mengganggu fitur atau operasional sebagian. |
| 3 | Tinggi | Menghambat fitur utama, keamanan, data, atau go-live. |

---

## 3.3 Skor Risiko

Skor Risiko \= Kemungkinan × Dampak

| Skor | Level Risiko | Tindakan |
| ----: | :---- | :---- |
| 1–2 | Rendah | Monitor berkala. |
| 3–4 | Sedang | Buat mitigasi dan pantau. |
| 6 | Tinggi | Mitigasi wajib sebelum go-live/pilot. |
| 9 | Kritis | Hentikan pekerjaan/rollout sampai risiko ditangani. |

# 6\. Risiko Scope dan Product Management

## R-001 — Scope MVP Melebar ke Fitur Non-MVP

### Deskripsi

Pengembangan MVP berisiko melebar ke fitur besar seperti TTE penuh, SRIKANDI, SIMPEG, SIPD, e-Kinerja, SATUSEHAT Kemenkes, Khanza, mobile app, OCR, AI fraud detection, dan pembayaran otomatis.

### Dampak

1. MVP terlambat.

2. Fitur inti tidak selesai.

3. Kompleksitas meningkat.

4. Risiko teknis dan regulasi meningkat.

5. UAT sulit dilakukan.

### Indikator Peringatan

1. Issue berisi kata “sekalian”.

2. AI coding agent membuat endpoint integrasi eksternal.

3. Sprint bertambah tanpa keputusan.

4. Database mulai memuat tabel non-MVP.

5. Diskusi pilot berubah menjadi pengembangan fase 2\.

### Mitigasi

1. Gunakan MoSCoW.

2. Gunakan CHANGE\_CONTROL\_AND\_DECISION\_LOG.md.

3. Label fitur non-MVP sebagai priority:wont-mvp.

4. Masukkan usulan ke FUTURE\_ROADMAP\_AND\_PHASE\_2\_BACKLOG.md.

5. Product Owner memutuskan setiap perubahan mayor.

### Residual Risk

Sedang, karena kebutuhan stakeholder dapat berkembang saat pilot.

### Trigger Eskalasi

Jika ada permintaan fitur non-MVP yang memengaruhi sprint aktif.

---

## R-002 — Prioritas Sprint Tidak Diikuti

### Deskripsi

Developer atau AI coding agent melompat ke Sprint 3/4/5 sebelum Sprint 0/1/2 selesai.

### Dampak

1. Fondasi tidak stabil.

2. Permission belum siap.

3. Audit belum tersedia.

4. Data tidak konsisten.

5. Fitur sulit dites.

### Mitigasi

1. Terapkan SPRINT\_EXECUTION\_PLAN.md.

2. Gunakan sprint gate.

3. Jangan mulai sprint berikutnya sebelum Must Have selesai.

4. Buat sprint review report.

5. Product Owner menahan perluasan scope.

### Trigger Eskalasi

Jika issue sprint berikutnya mulai dikerjakan tanpa gate selesai.

# 7\. Risiko Arsitektur dan Repository

## R-003 — Perubahan Core AWCMS-Micro Tidak Terkendali

### Deskripsi

Implementasi fitur daerah dilakukan dengan mengubah core AWCMS-Micro, bukan melalui plugin.

### Dampak

1. Sulit sinkron upstream.

2. Update AWCMS-Micro sulit.

3. Risiko regresi tinggi.

4. Plugin boundary rusak.

5. Maintenance jangka panjang berat.

### Mitigasi

1. Plugin-first architecture.

2. Core hanya diubah jika benar-benar perlu.

3. Setiap perubahan core wajib ADR.

4. AI coding agent wajib stop bila perlu core change.

5. Review khusus oleh admin teknis.

### Trigger Eskalasi

Jika file core harus diubah untuk memenuhi fitur daerah.

---

## R-004 — Plugin Boundary Tidak Jelas

### Deskripsi

Plugin membaca tabel plugin lain secara langsung tanpa service contract.

### Dampak

1. Coupling antar-plugin tinggi.

2. Perubahan satu plugin merusak plugin lain.

3. Dashboard sulit dikembangkan.

4. ABAC sulit diterapkan.

5. Test menjadi kompleks.

### Mitigasi

1. Gunakan PLUGIN\_ARCHITECTURE.md.

2. Gunakan API\_SERVICE\_CONTRACT.md.

3. Dashboard membaca aggregate service.

4. Cross-plugin data menggunakan snapshot/reference.

5. Review dependency antar-plugin.

### Trigger Eskalasi

Jika plugin baru membutuhkan query langsung ke tabel plugin lain.

Jika ada fitur delete permanen arsip final.

# 18\. Risiko Keamanan dan Privasi

## R-025 — Secret Bocor

### Deskripsi

Token, password, API key, private key, atau credential production masuk repository, log, issue, screenshot, atau dokumentasi.

### Dampak

1. Sistem dapat diakses pihak tidak berwenang.

2. Integrasi eksternal berisiko.

3. Harus rotasi credential.

4. Risiko insiden keamanan.

### Mitigasi

1. Jangan commit .env.

2. Gunakan .env.example.

3. Jangan tampilkan secret di output AI.

4. Secret scan manual.

5. Rotasi jika bocor.

6. Document only variable names.

### Trigger Eskalasi

Jika secret terlihat di commit/log/issue.

---

## R-026 — Upload File Berbahaya

### Deskripsi

User mengunggah file executable/script atau file dengan double extension.

### Dampak

1. Risiko keamanan server/storage.

2. Risiko malware.

3. Risiko akses file tidak sah.

4. Sistem perlu dibersihkan.

### Mitigasi

1. Extension whitelist.

2. MIME validation.

3. File size limit.

4. Private storage.

5. Rename/normalize filename.

6. Reject executable.

7. Upload attack test.

### Trigger Eskalasi

Jika file berbahaya berhasil diunggah.

---

## R-027 — Error Menampilkan Stack Trace atau Data Sensitif

### Deskripsi

Error API/UI menampilkan stack trace, query, path storage, atau data sensitif.

### Dampak

1. Informasi teknis bocor.

2. Risiko eksploitasi.

3. User melihat informasi yang tidak perlu.

4. Reputasi sistem turun.

### Mitigasi

1. Error response standar.

2. Hide stack trace in production.

3. Log internal terkontrol.

4. Sanitasi error.

5. Security testing.

### Trigger Eskalasi

Jika error production menampilkan detail teknis sensitif.

# 24\. Risk Response Strategy

## 24.1 Strategi Tanggapan

| Strategi | Keterangan | Contoh |
| :---- | :---- | :---- |
| Avoid | Menghindari risiko dengan tidak mengerjakan fitur | Menunda integrasi TTE dari MVP |
| Mitigate | Mengurangi kemungkinan/dampak | Backup sebelum migration |
| Transfer | Mengalihkan ke pihak/pemilik sistem terkait | Validasi integrasi eksternal oleh owner sistem |
| Accept | Menerima risiko dengan catatan | Bug low diterima saat pilot |
| Escalate | Menaikkan keputusan ke Product Owner/pimpinan | Perubahan workflow approval besar |

## 24.2 Risk Acceptance Rules

Risiko boleh diterima jika:

1. Level rendah atau sedang.

2. Tidak menyangkut data sensitif.

3. Tidak membuka data finance/restricted.

4. Tidak menghambat fitur utama.

5. Ada workaround.

6. Disetujui Product Owner.

7. Dicatat dalam decision log.

Risiko tidak boleh diterima jika:

1. Level kritis.

2. Menyangkut kebocoran data.

3. Menyangkut permission bypass.

4. Menyangkut dokumen final hilang/berubah tanpa audit.

5. Backup/restore gagal.

6. Bug critical belum selesai.

7. PDF tidak dapat dibuat sama sekali.

8. Approval tidak sah.

9. Data pasien masuk MMC.

10. Secret bocor.

# 29\. Go-Live Risk Gate

MVP tidak boleh go-live pilot jika:

* Ada bug critical terbuka.

* Permission bypass belum diperbaiki.

* ABAC belum berjalan.

* Dokumen finance terbuka ke user umum.

* PDF ST tidak dapat dibuat.

* Upload final tidak berjalan.

* Backup belum berhasil.

* Restore test belum berhasil.

* Audit trail aksi penting belum berjalan.

* User pilot belum dilatih.

* SOP pilot belum tersedia.

* Rollback plan belum tersedia.

* Dashboard data uji tidak valid.

* Security checklist belum lulus.

* Product Owner belum menyetujui.

---

# 30\. Risiko yang Tidak Boleh Ditangani Diam-Diam

Risiko berikut harus dicatat dan dilaporkan:

1. Data terlihat oleh user tidak berwenang.

2. Data finance terbuka.

3. Dokumen restricted dapat diunduh sembarang user.

4. Secret terlihat di log/commit.

5. Data pasien masuk MMC.

6. Migration gagal.

7. Backup gagal.

8. Restore gagal.

9. Approval salah tahap.

10. PDF final salah substansi.

11. Arsip final hilang.

12. Audit log tidak tercatat.

13. Upload file berbahaya lolos.

14. Dashboard menampilkan data salah untuk pimpinan.

15. AI coding agent mengubah core tanpa keputusan.

---

# 31\. Acceptance Criteria Pengelolaan Risiko

Pengelolaan risiko dianggap siap bila:

1. Risk register tersedia.

2. Pemilik risiko ditetapkan.

3. Risiko critical dan high memiliki mitigasi.

4. Stop condition AI coding agent jelas.

5. Risk escalation matrix tersedia.

6. Go-live risk gate tersedia.

7. Backup/restore risk dikendalikan.

8. Security/privacy risk dikendalikan.

9. Scope creep risk dikendalikan.

10. Risk review template tersedia.

11. Risk acceptance template tersedia.

12. Product Owner dapat mengambil keputusan berbasis risk register.
