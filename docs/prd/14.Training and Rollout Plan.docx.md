## Rencana Pelatihan dan Rollout MVP Satu Sehat Kobar v1.4

### Agenda Dinkes, ST/SPPD, Approval, PDF, Bukti Tugas, Jurnal Pegawai, Dashboard, MMC Terbatas, dan Arsip Digital Dasar

---

## 1\. Tujuan Dokumen

Dokumen ini menjadi panduan pelaksanaan pelatihan dan rollout MVP **Satu Sehat Kobar v1.4**.

Dokumen ini digunakan untuk memastikan:

1. User memahami tujuan sistem.

2. User memahami role masing-masing.

3. User dapat menggunakan fitur sesuai tugasnya.

4. Pilot berjalan terkendali.

5. Rollout tidak dilakukan sekaligus tanpa kesiapan.

6. Kendala user dapat dicatat dan ditindaklanjuti.

7. Tim support siap sebelum go-live.

8. Pimpinan mendapat gambaran progres implementasi.

9. Perluasan ke unit/faskes dilakukan bertahap.

10. Evaluasi pelatihan dan rollout terdokumentasi.

---

## 2\. Prinsip Pelatihan dan Rollout

Pelatihan dan rollout harus mengikuti prinsip berikut:

1. **Bertahap.**  
   Rollout dimulai dari Tim SIK dan unit pilot sebelum diperluas.

2. **Role-based training.**  
   Materi pelatihan disesuaikan dengan role pengguna.

3. **Praktik langsung.**  
   User tidak hanya mendengar penjelasan, tetapi mencoba membuat agenda, ST, approval, PDF, bukti, dan dashboard.

4. **Tidak melebar dari MVP.**  
   Pelatihan hanya mencakup fitur MVP, bukan integrasi fase berikutnya.

5. **Data uji aman.**  
   Pelatihan menggunakan data dummy atau data simulasi yang tidak sensitif.

6. **Support aktif.**  
   Setelah pelatihan harus tersedia kanal bantuan.

7. **Feedback dicatat.**  
   Masukan user dipilah menjadi bug, perbaikan minor, SOP, atau backlog fase berikutnya.

8. **Evaluasi terukur.**  
   Keberhasilan pelatihan dinilai dari kemampuan user menyelesaikan skenario.

9. **Dokumentasi diperbarui.**  
   Manual dan SOP disesuaikan setelah pelatihan/UAT.

10. **Perluasan hanya setelah pilot stabil.**  
    Semua unit/faskes tidak langsung menggunakan sistem sebelum hasil pilot dievaluasi.

# 5\. Strategi Rollout

## 5.1 Tahapan Rollout

Rollout dilakukan dalam 5 tahap:

Tahap 1: Internal Tim SIK  
Tahap 2: Unit Dinkes pilot  
Tahap 3: Faskes pilot  
Tahap 4: Perluasan internal Dinkes  
Tahap 5: Perluasan bertahap ke seluruh faskes

## 5.3 Tahap 1 — Training Tim SIK

Tujuan:

1. Tim SIK memahami seluruh fitur MVP.

2. Tim SIK siap menjadi trainer dan support.

3. Tim SIK memahami alur teknis dan operasional.

4. Tim SIK mampu mencatat bug dan feedback.

Peserta:

1. Ketua/Koordinator Tim SIK.

2. Admin teknis.

3. Support/helpdesk.

4. Dokumentator.

5. PIC UAT.

Output:

1. Tim SIK siap menjadi fasilitator.

2. Skenario demo siap.

3. Data dummy siap.

4. Manual awal siap.

5. Kanal support siap.

---

## 5.4 Tahap 2 — Training Unit Dinkes Pilot

Tujuan:

1. Menguji alur Agenda → ST/SPPD → Approval → PDF → Bukti.

2. Memastikan sekretariat, keuangan, operator surat, dan bidang teknis memahami perannya.

3. Mengidentifikasi gap SOP sebelum faskes dilibatkan.

Peserta:

1. Sekretariat.

2. Keuangan.

3. Operator surat.

4. Perwakilan bidang teknis.

5. Reviewer MMC.

6. Viewer pimpinan.

Output:

1. Alur internal Dinkes teruji.

2. Format PDF divalidasi awal.

3. Alur approval divalidasi awal.

4. Data finance/SPPD divalidasi awal.

5. Feedback internal dikumpulkan.

---

## 5.5 Tahap 3 — Training Faskes Pilot

Tujuan:

1. Memastikan faskes pilot memahami alur pengajuan dan approval.

2. Menguji ABAC faskes.

3. Menguji dashboard dan jurnal pada konteks faskes.

4. Mengidentifikasi kebutuhan pelatihan tambahan.

Peserta:

1. Kepala Faskes.

2. Kepala TU.

3. Admin faskes.

4. Operator surat faskes.

5. Pengelola program.

6. Pegawai pelaksana.

Output:

1. Faskes pilot siap menggunakan sistem.

2. User faskes memahami batas akses.

3. Alur ST faskes diuji.

4. Feedback faskes dikumpulkan.

---

## 5.6 Tahap 4 — Perluasan Internal Dinkes

Tujuan:

1. Memperluas penggunaan ke bidang/unit lain di Dinkes.

2. Menstandarkan penggunaan ST/SPPD berbasis sistem.

3. Mengurangi proses manual bertahap.

4. Memperkuat dashboard pimpinan.

Output:

1. Seluruh bidang Dinkes mulai menggunakan sistem.

2. SOP internal diperbarui.

3. Dashboard lebih representatif.

4. Kendala operasional terpetakan.

---

## 5.7 Tahap 5 — Perluasan Semua Faskes

Tujuan:

1. Memperluas penggunaan ke seluruh faskes secara bertahap.

2. Menyiapkan faskes champion.

3. Meningkatkan konsistensi data perjalanan dinas/pembinaan.

4. Menjadikan Satu Sehat Kobar sebagai platform kerja internal.

Output:

1. Faskes onboard bertahap.

2. Admin faskes aktif.

3. Data faskes masuk dashboard.

4. Support model lebih matang.

# 19\. Risiko Pelatihan dan Rollout

| :---- | :---- | :---- |

| Faskes bingung batas akses | Data tidak dipakai optimal | Pelatihan role faskes |
| Bug muncul saat pelatihan | Kepercayaan user turun | Catat bug, jelaskan status pilot |
| Scope melebar saat diskusi | MVP tertunda | Catat sebagai backlog fase berikutnya |

# 23\. Hal yang Tidak Boleh Dilakukan Saat Rollout

Selama rollout MVP, tidak boleh:

1. Menambahkan fitur non-MVP secara langsung.

2. Mengubah workflow approval tanpa keputusan.

3. Mengubah role besar tanpa validasi.

4. Membuka data finance ke user umum.

5. Menggunakan data pasien dalam pelatihan.

6. Menggunakan data rahasia dalam data dummy.

7. Menghapus data pilot tanpa backup.

8. Melakukan deployment besar saat pelatihan berlangsung.

9. Menyatakan sistem final sebelum evaluasi pilot.

10. Mengabaikan feedback user.

11. Mengabaikan bug critical.

12. Melakukan perluasan penuh tanpa pilot stabil.
