**Versi:** 1.0

**Fungsi:** Standar penulisan GitHub issue yang self-contained untuk dieksekusi AI model junior bertoken terbatas, plus indeks cakupan seluruh backlog.

---

## 1. Tujuan

Dokumen ini memastikan setiap unit kerja MVP menjadi **GitHub issue yang bisa diselesaikan tanpa membaca seluruh PRD**. Issue adalah satuan eksekusi; AI model junior memuat skill yang relevan, membaca issue + 1–2 section dokumen yang dikutip, lalu mengeksekusi.

Selaras dengan: `08.GITHUB_ISSUE_TEMPLATES.docx.md` (template & label), `02.IMPLEMENTATION_BACKLOG.docx.md` (sumber item), `24.TECHNICAL_IMPLEMENTATION_REFERENCES.md` (runtime), `12.CHANGE_CONTROL_AND_DECISION_LOG.docx.md` (DEC-019).

---

## 2. Prinsip Issue Siap-AI (token terbatas)

1. **Self-contained.** Embed fakta yang dibutuhkan; jangan minta model membaca 24 dokumen.
2. **Context Capsule di atas.** Setiap issue mengawali dengan kapsul invariant + rujuk issue pin **#11** (`[CAPSULE]`).
3. **Kutip maksimal 1–2 section dokumen + 1 skill.** Cantumkan "read ONLY these".
4. **Atomic.** 1 issue = 1 unit kohesif (≤ 1–3 hari kerja). Pecah jika lebih besar.
5. **AC testable + DoD eksplisit.** Termasuk migrasi idempoten, RBAC/ABAC, audit, test, lint.
6. **Pakai skill eksekusi** `.opencode/skills/sskobar-*` sesuai bagian.
7. **Label + milestone.** `ai-ready` wajib; label tipe/plugin/prioritas; milestone = Sprint.

## 3. Anatomi Issue (ringkas)

```
1. Context Capsule  → invariants + "read ONLY" + ref #11 & docs/prd/24 §4
2. Goal             → 1 kalimat + Backlog ID + Plugin + Milestone
3. References       → 1–2 doc section + 1 skill
4. Scope            → Data (ctx.db) / API / Service / UI — perubahan eksak
5. Acceptance       → AC testable
6. Definition of Done → migrasi idempoten, RBAC/ABAC, audit, test, register, lint
```

Template tersedia di `.github/ISSUE_TEMPLATE/feature-ai-ready.md` dan `ui-ux-ai-ready.md`.

## 5. Invariant Inti (juga di #11 dan doc 24 §4)

- Plugin **Native**: dir `awcms-micro-<key>`, npm `@awcms-micro/plugin-<key>`, register di template `astro.config.mjs` `plugins:[]` + protected path.
- Persistensi **D1 langsung via `ctx.db`** (Kysely); doc 04 = skema fisik; migrasi idempoten `<prefix>_migrations` di `install`/`activate`. `ctx.kv` = cache. (DEC-019)
- Envelope `{data,meta}` / `{data,pagination,meta}` / `{error,meta}`; pesan error Bahasa Indonesia; urutan handler auth→RBAC→ABAC→validation→service→audit→envelope.
- Setiap mutasi menulis audit event; RBAC/ABAC di service layer.
- Binding: D1 `DB`, R2 `MEDIA`, KV `SESSION`, Images `IMAGES`.

## 8. Cara Membuat Batch Issue Berikutnya

Untuk setiap epic `ready`:

1. Buka `docs/prd/02.IMPLEMENTATION_BACKLOG.docx.md`, ambil tiap item `EPIC-NN-NNN` (User Story + Acceptance Criteria).
2. Pakai template `.github/ISSUE_TEMPLATE/feature-ai-ready.md` (atau `ui-ux-ai-ready.md`).
3. Isi: Goal (+Backlog ID), References (1–2 section + skill), Scope (tabel doc 04 + endpoint doc 05), AC (dari doc 02), DoD.
4. Label: `feat` + `ai-ready` + `plugin:<...>` + `priority:<...>` (+ `gap-resolution`/`security`/`db-migration` bila relevan). Milestone = Sprint epic.
5. Tautkan ke tracker epic (sebut `#<tracker>` di body).

Perintah contoh:

```bash
gh issue create --title "[FEAT] <judul> — <plugin> (EPIC-NN-NNN)" \
  --label feat --label ai-ready --label plugin:<x> --label priority:high \
  --milestone "Sprint N" --body-file /tmp/issue.md
```

> Konsistensi: nama tabel ikuti `docs/prd/04` (mis. agenda = `agenda_events`, bukan `satusehat_agenda` di backlog — lihat tabel gap doc 24 §8).
