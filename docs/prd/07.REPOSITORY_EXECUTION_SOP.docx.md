# REPOSITORY\_EXECUTION\_SOP.md

## SOP Eksekusi Repository untuk Implementasi MVP Satu Sehat Kobar v1.4

### Panduan Kerja AI Coding Agent, Developer, dan Reviewer

---

## 1\. Tujuan Dokumen

Dokumen ini adalah SOP teknis untuk menjalankan pekerjaan implementasi MVP **Satu Sehat Kobar v1.4** langsung di repository berbasis **AWCMS-Micro**.

SOP ini digunakan untuk memastikan setiap AI coding agent atau developer:

1. Tidak langsung mengubah kode tanpa membaca repository.

2. Tidak mengubah core AWCMS-Micro tanpa alasan kuat.

3. Tidak membuat fitur di luar MVP.

4. Tidak melewati audit, permission, test, dan dokumentasi.

5. Tidak membocorkan secret.

6. Tidak membuat perubahan besar tanpa issue.

7. Tidak menjalankan deployment tanpa backup dan validasi.

8. Bekerja secara atomic, aman, dan dapat ditinjau.

---

## 2\. Prinsip Kerja Utama

Setiap pekerjaan di repository wajib mengikuti prinsip:

1. **Read first** — baca repository sebelum mengubah kode.

2. **Understand before edit** — pahami pola existing sebelum membuat struktur baru.

3. **Plugin-first** — fitur Satu Sehat Kobar dibuat melalui plugin/template/service/migration, bukan core.

4. **Atomic change** — satu perubahan harus kecil dan dapat diuji.

5. **Issue-driven** — pekerjaan harus terkait issue/milestone bila menggunakan GitHub.

6. **Test before done** — jangan menyatakan selesai tanpa validasi.

7. **Docs always updated** — dokumentasi diperbarui bila perilaku sistem berubah.

8. **No secret exposure** — jangan tampilkan atau commit secret.

9. **Rollback-aware** — perubahan database/deployment harus punya rencana rollback.

10. **Scope locked** — jangan mengerjakan fitur di luar MVP tanpa keputusan eksplisit.

\# 6\. Aturan Menentukan Atomic Slice

Sebelum mengubah kode, tentukan atomic slice.

\#\# 6.1 Atomic Slice yang Baik

Atomic slice yang baik memiliki ciri:

1\. Tujuan kecil.  
2\. Scope jelas.  
3\. File terdampak terbatas.  
4\. Dapat diuji.  
5\. Tidak mencampur banyak modul.  
6\. Tidak melebar ke fitur lain.  
7\. Tidak menyebabkan migration besar tanpa kebutuhan.  
8\. Tidak menyentuh core tanpa alasan.  
9\. Dapat di-commit dengan satu pesan jelas.  
10\. Dapat di-review cepat.

\#\# 6.2 Contoh Atomic Slice

\`\`\`text id="hpib9k"  
feat(platform): add satusehat plugin registry table  
feat(platform): seed satusehat plugin registry  
feat(agenda): add agenda events migration  
feat(agenda): add agenda list API  
feat(agenda): add create agenda form  
feat(duty): add duty requests migration  
feat(duty): add duty request draft service  
feat(approval): add duty approval table  
feat(pdf): add ST template seed  
feat(evidence): add evidence upload validation  
docs(satusehat): add repository audit  
test(agenda): add agenda validation tests

## 6.3 Atomic Slice yang Buruk

Contoh yang harus dihindari:

text id="xwium5" Implement all Satu Sehat Kobar features Add agenda, duty, PDF, dashboard, UAT, deployment Refactor core and add new plugin system Update everything Fix all bugs

# 7\. Workflow Implementasi Issue

Setiap issue harus dikerjakan dengan alur berikut.

## 7.1 Sebelum Coding

* Baca issue.

* Pastikan issue berada di milestone yang benar.

* Pastikan scope jelas.

* Pastikan acceptance criteria jelas.

* Pastikan dependency issue diketahui.

* Pastikan tidak masuk fitur Won’t Have MVP.

* Pastikan file yang akan diubah sudah dipahami.

* Pastikan command test diketahui.

* Pastikan tidak perlu credential yang belum tersedia.

* Pastikan risiko data diketahui.

## 7.2 Saat Coding

* Ikuti pola existing repository.

* Jangan membuat pola baru bila pola existing cukup.

* Jangan mengubah core tanpa alasan kuat.

* Buat migration bila perlu.

* Buat service/handler.

* Buat route/API.

* Terapkan permission.

* Terapkan ABAC bila data lintas unit/faskes.

* Terapkan audit trail.

* Buat UI sesuai pola existing.

* Tambahkan validasi input.

* Tambahkan error handling.

* Tambahkan test bila memungkinkan.

* Update dokumentasi.

## 7.3 Setelah Coding

* Jalankan format/lint.

* Jalankan typecheck bila tersedia.

* Jalankan unit test bila tersedia.

* Jalankan integration test bila tersedia.

* Jalankan build.

* Jalankan migration pada environment uji.

* Jalankan smoke test fitur.

* Cek tidak ada secret ter-commit.

* Cek perubahan file.

* Update changelog.

* Commit atomic.

* Laporkan hasil.

5. Jangan menggunakan branch production untuk eksperimen.

# 20\. Stop Conditions

AI coding agent atau developer harus berhenti dan melapor bila menemukan:

1. Perlu mengubah core AWCMS-Micro.

2. Migration berpotensi menghapus data.

3. Test utama gagal dan penyebab belum jelas.

4. Ada risiko kebocoran data.

5. Ada secret terbaca di log/output.

6. Ada perubahan scope ke fitur non-MVP.

7. Ada kebutuhan integrasi eksternal besar.

8. Ada konflik dependency besar.

9. Ada permission bypass.

10. Ada akses dokumen lintas unit tanpa izin.

11. Format tata naskah belum jelas.

12. Aturan SPPD/keuangan belum jelas.

13. Repository berbeda jauh dari asumsi PRD.

14. Perlu credential yang tidak tersedia.

15. Deployment berisiko tanpa rollback.
