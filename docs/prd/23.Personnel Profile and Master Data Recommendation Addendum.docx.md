# PERSONNEL\_PROFILE\_AND\_MASTER\_DATA\_RECOMMENDATION\_ADDENDUM.md

## Addendum Rekomendasi Master Data, Profil Pegawai, dan Integrasi User EmDash

### Tambahan untuk PRD dan Paket Dokumen MVP Satu Sehat Kobar v1.4 Berbasis AWCMS-Micro

---

## 1\. Status Dokumen

Dokumen ini merupakan addendum lanjutan untuk memperjelas kebutuhan master data yang sebelumnya masih tersirat.

Addendum ini secara khusus menegaskan kebutuhan:

1. Master unit organisasi Dinkes/faskes.

2. Master pegawai lokal.

3. Integrasi master pegawai dengan user EmDash.

4. Profil pegawai per orang.

5. Snapshot pegawai untuk dokumen ST/SPPD.

6. Master pejabat penandatangan.

7. Master penomoran naskah dinas.

8. Master dasar ST/jenis penugasan.

9. Master SPM/indikator/program.

10. Master hari libur/cuti bersama.

11. Master transport/alat angkut.

12. Master template approval chain.

---

## 2\. Kesimpulan Awal

Pada dokumen sebelumnya, data pegawai sudah masuk dalam bentuk awal melalui:

awcms-micro-personnel-master  
personnel.user\_id  
personnel\_snapshots

Namun, kebutuhan tersebut masih perlu diperjelas karena belum cukup menjelaskan:

1. Relasi formal antara user EmDash dan pegawai.

2. Profil pegawai per individu.

3. Data jabatan, unit kerja, dan faskes aktif.

4. Data pegawai yang digunakan pada ST/SPPD.

5. Data pegawai yang digunakan sebagai approver.

6. Data pegawai yang digunakan sebagai pejabat penandatangan.

7. Data pegawai yang digunakan pada jurnal pegawai.

8. Data pegawai yang dapat dikelola sendiri oleh user.

9. Data pegawai yang hanya boleh dikelola admin.

10. Data pegawai yang harus disimpan sebagai snapshot pada dokumen final.

Karena itu, perlu ditambahkan plugin dan schema yang lebih eksplisit untuk **Personnel Master dan Personnel Profile**.

# 6\. Plugin Master Pegawai Lokal

## 6.1 Nama Plugin

awcms-micro-personnel-master

## 6.2 Tujuan

Plugin ini mengelola master pegawai lokal sebagai referensi operasional sebelum integrasi SIMPEG tersedia.

Master pegawai digunakan untuk:

1. Pemohon ST/SPPD.

2. Pelaksana ST/SPPD.

3. Approver.

4. Verifikator bukti.

5. Reviewer MMC.

6. Operator surat.

7. Pejabat penandatangan.

8. Jurnal pegawai.

9. Dashboard pegawai.

10. Snapshot dokumen.

---

## 6.3 Relasi dengan User EmDash

Master pegawai harus dapat terhubung dengan user EmDash.

Prinsip relasi:

1. Satu user EmDash dapat dikaitkan dengan satu pegawai aktif.

2. Satu pegawai dapat memiliki satu user login aktif.

3. Pegawai dapat dibuat terlebih dahulu tanpa user login.

4. User EmDash dapat dibuat terlebih dahulu lalu dikaitkan ke pegawai.

5. Relasi user dan pegawai harus diaudit.

6. Jika user dinonaktifkan, data pegawai tidak otomatis dihapus.

7. Jika pegawai nonaktif, user dapat dinonaktifkan sesuai kebijakan admin.

8. Role tetap dikelola melalui sistem user/role EmDash.

9. Atribut pegawai digunakan untuk ABAC.

10. Snapshot pegawai tetap disimpan pada dokumen final.

## 6.5 Aturan Data Pegawai

1. full\_name wajib.

2. organization\_unit\_id wajib untuk pegawai aktif.

3. emdash\_user\_id tidak wajib saat pegawai dibuat.

4. Pegawai yang menjadi user aplikasi harus dikaitkan dengan emdash\_user\_id.

5. Pegawai pelaksana ST harus berasal dari master pegawai.

6. Approver harus berasal dari user EmDash yang terhubung dengan pegawai atau role resmi.

7. Pejabat penandatangan harus berasal dari master pegawai atau data pejabat manual yang tervalidasi.

8. Data pegawai nonaktif tidak boleh dipakai untuk ST baru kecuali role khusus.

9. Perubahan data pegawai penting harus tercatat audit.

10. Data pegawai pada dokumen final harus disimpan sebagai snapshot.

# 8\. Snapshot Pegawai untuk ST/SPPD

## 8.1 Tujuan Snapshot

Snapshot pegawai diperlukan agar dokumen ST/SPPD final tetap historis.

Contoh masalah tanpa snapshot:

1. Pegawai pindah unit setelah ST final.

2. Jabatan pegawai berubah setelah dokumen diterbitkan.

3. Pejabat penandatangan berubah.

4. Nama unit berubah.

5. Pangkat/golongan diperbarui.

6. Data dokumen lama ikut berubah bila hanya membaca master terbaru.

Karena itu, saat ST/SPPD final, sistem harus menyimpan snapshot.

## 8.3 Kapan Snapshot Dibuat

Snapshot dibuat saat:

1. Pengajuan ST/SPPD disubmit.

2. Pengajuan final approved.

3. Dokumen PDF generated.

4. Dokumen final uploaded.

5. Pegawai menjadi approver.

6. Pegawai menjadi pejabat penandatangan.

7. Pegawai menjadi pelaksana tugas.

8. Pegawai menjadi verifikator bukti.

9. Pegawai menjadi reviewer MMC bila diperlukan.

Untuk MVP, minimal snapshot wajib pada:

1. Pelaksana ST/SPPD.

2. Pejabat penandatangan.

3. Approver.

4. Unit organisasi pada dokumen final.

## 9.3 Aturan Integrasi

2. Role dan permission tetap mengikuti sistem user/role EmDash/AWCMS-Micro.

3. Data pegawai menjadi profil domain Satu Sehat Kobar.

4. emdash\_user\_id menjadi foreign reference dari pegawai ke user.

5. Pegawai dapat belum memiliki user login.

6. User login dapat belum memiliki profil pegawai, tetapi akan dibatasi akses domain.

7. Fitur ST/SPPD sebaiknya mensyaratkan user memiliki personnel profile aktif.

8. Admin harus dapat menghubungkan user ke pegawai.

9. Relasi user–pegawai harus memiliki audit.

10. Jika relasi berubah, dokumen final lama tidak berubah karena memakai snapshot.
