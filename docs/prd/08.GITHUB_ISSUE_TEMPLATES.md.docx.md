# GITHUB\_ISSUE\_TEMPLATES.md

## Template GitHub Issue untuk Implementasi MVP Satu Sehat Kobar v1.4

### Standar Issue Atomic, Traceable, dan Tidak Melebar

---

## 1\. Tujuan Dokumen

Dokumen ini berisi standar dan template GitHub Issue untuk implementasi MVP **Satu Sehat Kobar v1.4** berbasis **AWCMS-Micro**.

Dokumen ini digunakan agar setiap issue:

1. Memiliki tujuan yang jelas.

2. Memiliki scope yang terkendali.

3. Tidak melebar ke fitur non-MVP.

4. Memiliki acceptance criteria.

5. Memiliki validasi teknis.

6. Memiliki risiko yang dicatat.

7. Memiliki kaitan dengan sprint, epic, dan milestone.

8. Dapat dikerjakan oleh AI coding agent secara atomic.

9. Dapat direview oleh Tim SIK/developer.

10. Dapat dilacak sampai UAT dan go-live pilot.

---

## 2\. Prinsip Issue

Setiap issue harus mengikuti prinsip:

1. **Satu issue \= satu fokus pekerjaan.**

2. **Satu issue harus bisa diuji.**

3. **Satu issue harus punya acceptance criteria.**

4. **Satu issue tidak boleh memuat fitur di luar MVP.**

5. **Satu issue tidak boleh mencampur terlalu banyak modul.**

6. **Satu issue harus menyebut area terdampak.**

7. **Satu issue harus menyebut risiko.**

8. **Satu issue harus menyebut validasi/test.**

9. **Satu issue harus menyebut dokumentasi yang perlu diperbarui.**

10. **Issue besar wajib dipecah menjadi beberapa issue atomic.**

---

## 3\. Milestone Utama

Semua issue MVP harus masuk milestone:

MVP Satu Sehat Kobar v1.4

Milestone ini hanya mencakup alur:

Agenda  
→ ST/SPPD  
→ Approval  
→ PDF  
→ Upload Final  
→ Bukti  
→ Verifikasi  
→ Jurnal  
→ Dashboard  
→ Arsip

Fitur di luar alur tersebut harus masuk backlog fase berikutnya, bukan MVP.

# 17\. Issue Quality Checklist

Sebelum issue dikerjakan, pastikan:

* Judul jelas.

* Epic jelas.

* Sprint jelas.

* Priority jelas.

* Scope jelas.

* Di luar scope jelas.

* Acceptance criteria tersedia.

* Validasi/test tersedia.

* Area terdampak jelas.

* Risiko dicatat.

* Tidak melebar ke fitur non-MVP.

* Tidak membutuhkan keputusan yang belum ada.

* Tidak membutuhkan credential yang belum tersedia.

* Tidak mengharuskan perubahan core tanpa alasan.

---

# 18\. Issue Splitting Rules

Issue harus dipecah bila:

1. Menyentuh lebih dari satu plugin besar.

2. Mengubah database dan UI besar sekaligus.

3. Mengubah workflow approval dan PDF sekaligus.

4. Mengubah permission lintas modul.

5. Membutuhkan lebih dari satu migration besar.

6. Tidak dapat diuji dalam satu alur sederhana.

7. Membutuhkan keputusan Product Owner.

8. Mengandung fitur MVP dan non-MVP sekaligus.

9. Berpotensi mengubah core.

10. Sulit direview dalam satu pull request.

Contoh pemecahan:

Buruk:  
Implement Plugin Agenda

Baik:  
S2-001 Buat struktur plugin Agenda  
S2-002 Migration agenda\_events  
S2-003 API CRUD Agenda  
S2-004 UI daftar dan form Agenda  
S2-005 Permission dan audit Agenda  
S2-006 API Agenda untuk ST/SPPD

---

# 19\. Issue Closing Criteria

Issue hanya boleh ditutup bila:

* Acceptance criteria terpenuhi.

* Validasi/test dijalankan.

* Dokumentasi diperbarui.

* Changelog diperbarui bila relevan.

* Tidak ada bug critical.

* Tidak ada permission bypass.

* Tidak ada secret.

* Tidak ada perubahan di luar scope.

* Review selesai.

* Commit/PR terkait jelas.
