# GitHub Issue Templates — Satu Sehat Kobar v1.5

## Template Standar GitHub Issue untuk Implementasi MVP

**Platform**: AWCMS-Micro / Cloudflare Workers  
**Versi Dokumen**: v1.5 (selaras dengan PRD v1.5)  
**Referensi**: Dokumen PRD, Implementation Backlog (doc 02), Sprint Execution Plan (doc 09)

---

## 1. Tujuan Dokumen

Dokumen ini berisi template GitHub Issue standar untuk implementasi **Satu Sehat Kobar**. Semua issue wajib menggunakan template ini agar:

1. Memiliki tujuan yang jelas dan scope yang terkendali
2. Tidak melebar ke fitur non-MVP tanpa persetujuan PO
3. Memiliki acceptance criteria yang dapat diuji
4. Mencatat risiko dan ketergantungan teknis
5. Dapat dikerjakan secara atomic oleh developer atau AI coding agent
6. Dapat direview oleh Tim SIK dengan kriteria yang jelas
7. Terlacak hingga UAT dan go-live pilot

---

## 2. Prinsip Issue

| Prinsip | Aturan |
|---------|--------|
| **Satu fokus** | Satu issue = satu unit kerja yang kohesif |
| **Dapat diuji** | Setiap issue harus memiliki acceptance criteria yang testable |
| **Atomic** | Issue harus selesai dalam 1-3 hari kerja; jika lebih, pecah menjadi sub-issue |
| **Scope terkendali** | Tidak mencampur fitur berbeda modul dalam satu issue |
| **Terlacak** | Issue ditautkan ke Epic, Sprint Milestone, dan PRD section terkait |
| **Aman** | Setiap issue yang menyentuh auth/RBAC/ABAC wajib sertakan security checklist |

## 3. Label Standar

### Label Tipe
- `feat` — Fitur baru
- `fix` — Perbaikan bug
- `chore` — Perubahan non-fungsional (docs, config, refactor)
- `test` — Penambahan atau perbaikan tes
- `security` — Issue keamanan
- `db-migration` — Perubahan schema database

### Label Plugin
- `plugin:agenda` — Plugin Agenda Dinkes
- `plugin:duty` — Plugin ST/SPPD
- `plugin:dashboard` — Plugin Dashboard
- `plugin:archive` — Plugin Arsip Dokumen
- `plugin:template` — Plugin Template Dokumen
- `plugin:mmc` — Plugin MMC Publikasi
- `plugin:spm` — Plugin SPM Health
- `platform:core` — Platform Core (AWCMS-Micro)

### Label Prioritas
- `priority:critical` — Blocker untuk sprint
- `priority:high` — Harus selesai dalam sprint ini
- `priority:medium` — Target sprint ini, bisa pindah
- `priority:low` — Nice to have

### Label Status
- `status:ready` — Siap dikerjakan
- `status:in-progress` — Sedang dikerjakan
- `status:blocked` — Terblokir (sertakan alasan)
- `status:review` — Menunggu code review
- `status:testing` — Dalam testing
- `status:done` — Selesai dan merged

### Label Gap Resolution
- `gap-resolution` — Issue yang menyelesaikan gap dari PRD v1.5

## 4. Template Issue: Fitur Baru (Feature)

```markdown
## [FEAT] [Nama Fitur] — [Plugin/Modul]

### Konteks & Tujuan
<!-- Jelaskan MENGAPA fitur ini diperlukan. Referensikan PRD section jika ada. -->
<!-- Contoh: Implementasi PRD §F02.3 — SPPD: rincian anggaran per baris dengan budget_category -->

**PRD Reference**: §[X.X] — [Judul Section]
**Epic**: EPIC-[##] — [Nama Epic]
**Sprint Target**: Sprint [#]
**Plugin**: [nama plugin]

### Deskripsi Teknis
<!-- Jelaskan APA yang harus diimplementasikan. Sertakan:
- Table/field yang perlu dibuat atau dimodifikasi
- Endpoint API yang perlu dibuat
- Komponen UI yang perlu dibuat (jika ada)
- Service contract yang perlu diimplementasikan -->

**Database Changes**:
- [ ] Tabel baru: `[nama_tabel]` dengan field [field1], [field2], ...
- [ ] Modifikasi tabel: `[nama_tabel]` tambah field `[field_name]`
- [ ] Migration file: `[YYYYMMDDHHMMSS]_[deskripsi].sql`

**API Changes**:
- [ ] `[METHOD] /api/[path]` — [deskripsi singkat]
- Auth: [required/tidak]
- RBAC: permission `[permission.name]`
- ABAC: [aturan ABAC yang berlaku]

**UI Changes** (jika ada):
- [ ] Halaman/komponen: [nama]
- [ ] Form fields: [field list]

### Acceptance Criteria
<!-- Kriteria yang harus terpenuhi agar issue dianggap selesai -->
- [ ] [AC-1] [Deskripsi kriteria yang spesifik dan testable]
- [ ] [AC-2] [Deskripsi kriteria]
- [ ] [AC-3] Audit event `[event.type]` dicatat ke `satusehat_audit_logs`
- [ ] [AC-4] Error response menggunakan format standar dengan pesan Bahasa Indonesia
- [ ] [AC-5] Unit test untuk service layer mencakup happy path dan edge cases

### RBAC / ABAC Checklist
- [ ] Permission `[permission.name]` diverifikasi di route handler
- [ ] ABAC rule: [deskripsi aturan ABAC spesifik]
- [ ] Tidak ada data lintas unit/faskes yang bocor

### Security Checklist
- [ ] Input divalidasi dengan Zod schema
- [ ] Query menggunakan parameterized queries (tidak ada string concatenation)
- [ ] File upload (jika ada): validasi MIME type + ukuran
- [ ] Tidak ada secrets hardcoded

### Risiko & Ketergantungan
- **Ketergantungan**: [Issue/PR lain yang harus selesai lebih dulu]
- **Risiko**: [Risiko teknis yang perlu diwaspadai]
- **Rollback**: [Cara rollback jika terjadi masalah di production]

### Definition of Done
- [ ] Kode lulus code review oleh minimal 1 reviewer
- [ ] Semua acceptance criteria terpenuhi
- [ ] Unit test passing (min 70% coverage service layer)
- [ ] Tidak ada critical/high severity lint errors
- [ ] Migration file sudah ditest di staging
- [ ] Dokumentasi API diperbarui (jika ada endpoint baru)
```

## 5. Template Issue: Bug Fix

```markdown
## [BUG] [Deskripsi Bug] — [Plugin/Modul]

### Deskripsi Bug
<!-- Jelaskan bug secara singkat dan jelas -->

**Severity**: Critical / High / Medium / Low
**Plugin**: [nama plugin]
**Sprint Target**: Sprint [#]

### Langkah Reproduksi
1. [Step 1]
2. [Step 2]
3. ...

**Expected behavior**: [Apa yang seharusnya terjadi]
**Actual behavior**: [Apa yang sebenarnya terjadi]
**Environment**: [development / staging / production]

### Root Cause Analysis
<!-- Diisi setelah investigasi awal -->
- Kemungkinan penyebab: [deskripsi]
- File yang terdampak: [path/file]

### Fix Plan
- [ ] [Perubahan yang perlu dilakukan]
- [ ] Tambahkan regression test untuk memastikan bug tidak berulang

### Acceptance Criteria
- [ ] Bug tidak lagi dapat direproduksi dengan langkah di atas
- [ ] Regression test ditambahkan
- [ ] Tidak ada perubahan perilaku fitur lain

### Dampak
- [ ] Tidak ada breaking change pada API
- [ ] Tidak ada perubahan schema database
- [ ] Data yang sudah ada tidak terdampak
```

## 6. Template Issue: Database Migration

```markdown
## [DB-MIGRATION] [Deskripsi Perubahan Schema] — [Plugin/Modul]

### Tujuan Migration
<!-- Jelaskan MENGAPA schema perlu diubah -->

**Plugin**: [nama plugin]
**Prefix Tabel**: `[prefix]_`
**Sprint Target**: Sprint [#]

### Perubahan Schema
```sql
-- Migration: [YYYYMMDDHHMMSS]_[deskripsi].sql
-- Plugin: [nama plugin]

-- UP (apply changes)
[SQL statements]

-- DOWN (rollback — jika memungkinkan)
[SQL statements]
```

### Data Migration (jika ada)
<!-- Apakah ada data yang perlu dimigrasikan? -->
- [ ] Tidak ada data migration
- [ ] Ada data migration: [deskripsi transformasi data]

### Dampak
- [ ] Jumlah tabel terdampak: [#]
- [ ] Jumlah baris data yang mungkin terdampak: [#]
- [ ] Breaking change untuk plugin lain: Ya / Tidak

### Acceptance Criteria
- [ ] Migration berjalan tanpa error di D1 staging
- [ ] Rollback migration berjalan tanpa error
- [ ] Data yang sudah ada tidak terdampak negatif
- [ ] Index yang diperlukan sudah ditambahkan
- [ ] Sesuai prefix tabel plugin (`[prefix]_`)

### Pre-Migration Checklist
- [ ] Backup database staging sudah dilakukan
- [ ] Migration sudah di-review oleh Tech Lead
- [ ] Tidak ada direct reference ke tabel plugin lain
```

## 8. Contoh Issue Nyata

### Contoh 1: Implementasi duty_approval_step_config (GAP-01)

```markdown
## [GAP-RESOLUTION] GAP-01 — Approval Chain Configuration Table

**Gap ID**: GAP-01
**Deskripsi Gap**: Approval chain sebelumnya bersifat hardcoded; tidak ada cara konfigurasi per org_level
**Keputusan**: DEC-005 — Approval chain dikonfigurasi di DB (duty_approval_step_config)
**PRD v1.5 Reference**: §11.4 ABAC Rules

### Implementasi

**Database Changes**:
- [ ] Buat tabel `duty_approval_step_config` dengan field:
  - id, org_level (dinas|faskes), step_order, approval_step, approver_role, is_optional, skip_condition

**API Changes**:
- [ ] GET `/api/duty-requests/:id/approval-chain` — return step config untuk request ini
- [ ] Logic: load config berdasarkan org_level, set finance step status='skipped' jika is_budgeted=false

**Business Logic**:
- [ ] Saat ST submitted, buat duty_request_approvals berdasarkan config
- [ ] Finance step: jika is_budgeted=false → set status='skipped' otomatis

### Acceptance Criteria
- [ ] Tabel duty_approval_step_config dapat dikonfigurasi via admin UI atau seed
- [ ] ST tidak berbiaya: finance step status='skipped' otomatis, tidak muncul di antrian Keuangan
- [ ] ST berbiaya: finance step muncul setelah secretary_verified
- [ ] Config berbeda untuk org_level='dinas' vs 'faskes'
- [ ] Unit test: 3 skenario (dinas berbiaya, dinas tidak berbiaya, faskes)
```

### Contoh 2: Dashboard KV Caching (GAP-06)

```markdown
## [GAP-RESOLUTION] GAP-06 — Dashboard KV Caching 15 Menit

**Gap ID**: GAP-06
**Keputusan**: DEC-014 — Dashboard aggregate di-cache di KV dengan TTL 15 menit
**PRD v1.5 Reference**: §7 Dashboard F07.5

### Implementasi

**KV Cache Keys**:
- `dashboard:aggregate:{user_unit_id}` — KPI cards per unit
- `dashboard:spm:{period}` — SPM summary per periode
- `dashboard:overview` — Global overview (admin only)

**Business Logic**:
- [ ] Setiap GET /api/satusehat/dashboard: cek KV dulu, jika hit return cache
- [ ] Cache miss: query D1, simpan ke KV dengan TTL=900 (15 menit)
- [ ] Cache invalidation: saat ada event mutasi (duty status change, evidence verified)

### Acceptance Criteria
- [ ] Dashboard load <3 detik (cache hit: <500ms)
- [ ] Cache di-invalidate saat ada perubahan data relevan
- [ ] Cache key menyertakan unit/faskes scope (tidak bocor antar unit)
- [ ] KV key pattern terdokumentasi
```

## 9. Konvensi Penamaan Issue

**Format Judul**: `[TYPE] [Plugin/Scope] — [Deskripsi singkat aksi]`

**Contoh**:
- `[FEAT] duty-travel — Implementasi wizard ST langkah 1: data dasar`
- `[BUG] agenda — Kalender tidak menampilkan agenda bulan berikutnya`
- `[DB-MIGRATION] duty-travel — Tambah field budget_category ke duty_request_budget_lines`
- `[GAP-RESOLUTION] GAP-01 — duty_approval_step_config table dan API`
- `[CHORE] docs — Perbarui API documentation setelah endpoint baru`
