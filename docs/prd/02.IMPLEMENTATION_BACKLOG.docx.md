## 1. Konvensi Backlog

### 1.1 Format Item Backlog

Setiap item backlog mengikuti format standar berikut:

```text
[EPIC-##-###] Judul Item
Priority  : Must Have | Should Have | Could Have | Won't Have (MVP)
Effort    : S (≤1 hari) | M (2–3 hari) | L (4–7 hari) | XL (>7 hari)
Sprint    : Sprint 0 | Sprint 1 | … | Sprint 6
Plugin    : platform | agenda-dinkes | duty-travel | approval-workflow |
            document-template | document-archive | satusehat-dashboard |
            spm-health | mmc-publication | evidence | journal
```

### 1.2 Status Item

| Status       | Keterangan                                     |
|--------------|------------------------------------------------|
| `todo`       | Belum dimulai                                  |
| `in-progress`| Sedang dikerjakan di sprint aktif              |
| `review`     | PR dibuat, menunggu review                     |
| `done`       | Merged, acceptance criteria terpenuhi          |
| `blocked`    | Tertahan karena dependensi atau keputusan PO   |
| `deferred`   | Dipindah ke sprint berikutnya atau Phase 2     |

### EPIC-01: Platform Foundation

**Tujuan:** Menyiapkan infrastruktur teknis AWCMS-Micro, plugin registry, RBAC/ABAC baseline, dan audit log sebagai fondasi semua plugin lain.

---

**[EPIC-01-001] Setup Plugin Registry dan Loader**

- User Story: Sebagai Tech Lead, saya ingin semua plugin terdaftar di tabel `satusehat_plugin_registry` agar sistem dapat mengaktifkan/menonaktifkan plugin tanpa deploy ulang.
- Acceptance Criteria:
  1. Tabel `satusehat_plugin_registry` berhasil dibuat dengan kolom: `id`, `plugin_name`, `is_active`, `config_json`, `created_at`.
  2. Plugin loader membaca registry dari D1 saat startup Worker.
  3. Route plugin tidak ter-load jika `is_active = false`.
- Priority: Must Have
- Effort: M
- Sprint: Sprint 1

---

**[EPIC-01-002] Seed 17 Roles dan 30+ Permissions**

- User Story: Sebagai Admin, saya ingin semua role dan permission terisi secara otomatis via seed agar sistem siap digunakan tanpa konfigurasi manual.
- Acceptance Criteria:
  1. 17 role terdefinisi di tabel `roles` sesuai PRD (Kadis, Sekretaris, Kabid, dll.).
  2. 30+ permission terdefinisi di tabel `permissions`.
  3. Mapping role ↔ permission tersimpan di `role_permissions`.
- Priority: Must Have
- Effort: M
- Sprint: Sprint 1

---

**[EPIC-01-003] Admin Login dan Session Management**

- User Story: Sebagai Admin, saya ingin login dengan email/password dan mendapat session JWT agar bisa mengakses modul admin.
- Acceptance Criteria:
  1. Endpoint `POST /auth/login` mengembalikan JWT valid.
  2. JWT expired setelah 8 jam.
  3. Refresh token tersimpan di KV dengan TTL 7 hari.
- Priority: Must Have
- Effort: M
- Sprint: Sprint 1

---

**[EPIC-01-004] Middleware RBAC — Route Guard**

- User Story: Sebagai sistem, saya ingin setiap request ke endpoint terproteksi diperiksa role-nya agar tidak terjadi unauthorized access.
- Acceptance Criteria:
  1. Middleware memeriksa JWT dan mengekstrak `user_id` dan `role_ids`.
  2. Request tanpa permission yang sesuai mengembalikan HTTP 403.
  3. Audit log mencatat setiap access denied.
- Priority: Must Have
- Effort: M
- Sprint: Sprint 1

---

**[EPIC-01-005] Audit Log Infrastructure**

- User Story: Sebagai Auditor, saya ingin semua aksi penting tercatat di tabel audit log agar aktivitas sistem dapat ditelusuri.
- Acceptance Criteria:
  1. Tabel `satusehat_audit_logs` dibuat dengan kolom: `id`, `user_id`, `action`, `resource_type`, `resource_id`, `ip_address`, `timestamp`, `metadata_json`.
  2. Helper `auditLog(action, resource)` tersedia dan digunakan di semua handler.
  3. Query audit log tersedia di panel admin (filter by user, date, action).
- Priority: Must Have
- Effort: M
- Sprint: Sprint 1

---

**[EPIC-01-006] Setup duty_approval_step_config Table**

- User Story: Sebagai Admin, saya ingin konfigurasi langkah approval disimpan di database agar bisa dikustomisasi per org_level tanpa deploy ulang.
- Acceptance Criteria:
  1. Tabel `duty_approval_step_config` dibuat dengan kolom: `id`, `org_level`, `step_order`, `step_name`, `approver_role`, `is_mandatory`, `skip_condition`, `created_at`.
  2. Seed data default untuk org_level `dinas` dan `faskes`.
  3. API `GET /admin/approval-step-config` mengembalikan konfigurasi per org_level.
- Priority: Must Have
- Effort: M
- Sprint: Sprint 1
- Note: [GAP RESOLUTION — DEC-005]

---

**[EPIC-01-007] Health Check dan Monitoring Endpoint**

- User Story: Sebagai DevOps, saya ingin ada endpoint `/health` yang mengembalikan status semua dependensi agar monitoring otomatis dapat berjalan.
- Acceptance Criteria:
  1. `GET /health` mengembalikan status D1, KV, R2 dalam < 200ms.
  2. Jika salah satu dependensi down, response HTTP 503.
  3. Response mencakup: `{"status":"ok","db":"ok","kv":"ok","r2":"ok","version":"0.x.x"}`.
- Priority: Must Have
- Effort: S
- Sprint: Sprint 1
