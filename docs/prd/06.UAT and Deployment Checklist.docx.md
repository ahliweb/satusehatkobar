---

## 1\. Tujuan Dokumen

Dokumen ini menjadi panduan resmi untuk melakukan:

1. Validasi kesiapan teknis.

2. Validasi keamanan.

3. Validasi database dan migration.

4. Validasi fitur MVP.

5. Deployment ke staging/pilot.

6. Smoke test setelah deployment.

7. User Acceptance Test.

8. Backup dan restore test.

9. Rollback bila terjadi kegagalan.

10. Go-live pilot terbatas.

11. Evaluasi pasca go-live.

Dokumen ini wajib digunakan sebelum AI coding agent, developer, atau Tim SIK menyatakan MVP **Satu Sehat Kobar v1.4** siap digunakan secara operasional.

---

## 2\. Prinsip Utama

1. Tidak ada go-live tanpa backup.

2. Tidak ada go-live tanpa restore test.

3. Tidak ada go-live tanpa role dan permission diuji.

4. Tidak ada go-live tanpa UAT minimal.

5. Tidak ada go-live bila PDF ST/SPPD belum bisa dibuat.

6. Tidak ada go-live bila upload dokumen final belum diuji.

7. Tidak ada go-live bila bukti tugas belum bisa diunggah dan diverifikasi.

8. Tidak ada go-live bila dashboard belum menampilkan data benar.

9. Tidak ada go-live bila audit trail belum berjalan.

10. Tidak ada go-live bila masih ada bug critical.

11. Tidak ada deployment production/pilot tanpa rencana rollback.

12. Tidak ada perubahan scope saat UAT tanpa change request.

# 5\. Deployment Checklist

## 5.1 Pre-Deployment

* Tentukan environment target: staging atau pilot.

* Pastikan jadwal deployment disepakati.

* Informasikan kepada user pilot bila ada downtime.

* Pastikan backup database terbaru tersedia.

* Pastikan backup file/storage tersedia.

* Pastikan rollback plan tersedia.

* Pastikan environment variable tersedia.

* Pastikan secret tidak dicetak di log.

* Pastikan migration list diketahui.

* Pastikan build artifact siap.

* Pastikan PIC deployment tersedia.

* Pastikan PIC rollback tersedia.

---

## 5.2 Deployment Steps

Langkah deployment harus mengikuti pola repository. Secara umum:

text id="x9nh7o" 1\. Pull kode terbaru. 2\. Install/update dependency. 3\. Jalankan build bila diperlukan. 4\. Backup database. 5\. Backup storage/file penting. 6\. Jalankan migration. 7\. Jalankan seed yang diperlukan. 8\. Deploy aplikasi. 9\. Restart/reload service bila diperlukan. 10\. Jalankan smoke test. 11\. Cek log error. 12\. Catat hasil deployment.

Checklist:

* Kode terbaru sudah diambil.

* Dependency berhasil diinstall/update.

* Build berhasil.

* Backup sebelum deployment selesai.

* Migration berhasil.

* Seed berhasil.

* Aplikasi berhasil deploy.

* Service berhasil reload/restart.

* Smoke test berjalan.

* Tidak ada error critical di log.

* Hasil deployment dicatat.

# 10\. Bug Severity

## 10.1 Critical

Contoh:

1. Tidak bisa login.

2. Data user lain terbuka tanpa izin.

3. Dokumen finance terbuka untuk user biasa.

4. Generate PDF gagal total.

5. Upload final gagal total.

6. Bukti tidak bisa diverifikasi.

7. Data hilang.

8. Migration merusak data.

9. Secret bocor.

10. Backup/restore gagal.

Critical harus diselesaikan sebelum go-live.

---

## 10.2 High

Contoh:

1. Approval salah tahap.

2. Dashboard angka penting salah.

3. PDF format rusak tetapi masih bisa dibuat.

4. Filter data lintas unit tidak tepat.

5. Upload file valid sering gagal.

6. Jurnal tidak update otomatis.

7. Audit event penting tidak tercatat.

High sebaiknya diselesaikan sebelum go-live pilot, kecuali ada workaround resmi.

---

## 10.3 Medium

Contoh:

1. UI kurang rapi.

2. Label kurang jelas.

3. Validasi pesan kurang informatif.

4. Export belum sempurna.

5. Sorting/filter minor bermasalah.

6. Draft MMC perlu perbaikan redaksi.

Medium dapat masuk perbaikan sprint berikut bila tidak mengganggu alur utama.

---

## 10.4 Low

Contoh:

1. Typo.

2. Spasi/format minor.

3. Ikon kurang sesuai.

4. Empty state kurang informatif.

5. Warna badge perlu penyempurnaan.

Low tidak menghambat go-live pilot.

---

# 12\. Rollback Plan

## 12.1 Kondisi yang Memicu Rollback

Rollback dilakukan bila:

1. Login gagal untuk mayoritas user.

2. Data utama tidak bisa dibaca.

3. Migration merusak data.

4. Approval tidak berjalan.

5. Generate PDF gagal total.

6. Upload final gagal total.

7. Dokumen terbuka tanpa izin.

8. Secret terindikasi bocor.

9. Error critical berulang.

10. Backup/restore tidak valid.

11. Sistem tidak stabil untuk operasional pilot.

---

## 12.2 Langkah Rollback

Langkah umum:

text id="p3nsge" 1\. Hentikan akses user sementara bila diperlukan. 2\. Catat waktu dan penyebab rollback. 3\. Matikan fitur/plugin baru bila memungkinkan. 4\. Rollback build aplikasi ke versi sebelumnya. 5\. Restore database dari backup terakhir bila diperlukan. 6\. Restore storage/file bila diperlukan. 7\. Jalankan smoke test versi rollback. 8\. Informasikan status ke user pilot. 9\. Buat incident report. 10\. Buat rencana perbaikan.

Checklist:

* Akses user dihentikan sementara bila diperlukan.

* Penyebab rollback dicatat.

* Build versi sebelumnya tersedia.

* Backup database tersedia.

* Backup storage tersedia.

* Restore berhasil.

* Smoke test setelah rollback berhasil.

* User pilot diinformasikan.

* Incident report dibuat.
