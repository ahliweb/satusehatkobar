## 1. Pendekatan Keamanan (Security by Design)

Satu Sehat Kobar dibangun dengan pendekatan **Security by Design** — keamanan bukan fitur tambahan, melainkan bagian dari arsitektur inti sejak rancangan awal.

### 1.1 Prinsip Utama

| Prinsip | Implementasi |
|---|---|
| Least privilege | Setiap role hanya mendapat permission minimum yang dibutuhkan untuk tugasnya |
| Default deny | Semua akses ditolak kecuali secara eksplisit diizinkan (whitelist approach) |
| Need to know | Dokumen hanya dapat diakses oleh pihak yang memiliki kebutuhan nyata |
| Separation of duties | Pemohon, approver, keuangan, operator surat, auditor memiliki peran berbeda dan tidak tumpang tindih |
| Defense in depth | Keamanan berlapis: network → API → RBAC → ABAC → data → storage |
| Fail securely | Saat error/kegagalan, sistem menolak akses — bukan memberikan akses default |
| Immutable audit | Audit log tidak dapat diubah oleh user biasa maupun approver |
| No data by default | Tidak ada data sensitif yang tampil sebelum ada permission eksplisit |

### 1.2 Lingkup Keamanan MVP

Checklist ini mencakup 7 plugin MVP:
- `agenda-dinkes` — Agenda kegiatan Dinkes
- `duty-travel` — Pengajuan ST/SPPD dan bukti tugas
- `satusehat-dashboard` — Dashboard KPI dan SPM
- `spm-health` — Indikator Standar Pelayanan Minimal
- `mmc-publication` — Draft dan publikasi konten MMC
- `document-template` — Template dokumen resmi
- `document-archive` — Arsip digital dokumen final

### 1.3 Referensi Regulasi

- PP No. 95/2018 tentang Sistem Pemerintahan Berbasis Elektronik (SPBE)
- UU No. 27/2022 tentang Perlindungan Data Pribadi
- Permenkes No. 4/2019 tentang Standar Pelayanan Minimal Bidang Kesehatan
- Permendagri No. 35/2012 tentang Tata Naskah Dinas
- PP No. 28/2012 tentang Kearsipan
- ISO/IEC 27001:2022 sebagai referensi tata kelola keamanan informasi

## 2. Autentikasi

### 2.1 Checklist Implementasi Autentikasi

- [ ] HTTPS wajib aktif — semua HTTP redirect ke HTTPS (301)
- [ ] HSTS header terpasang: `Strict-Transport-Security: max-age=31536000; includeSubDomains`
- [ ] Session expire: **8 jam** setelah aktivitas terakhir
- [ ] CSRF protection aktif untuk semua endpoint yang menggunakan cookie session
- [ ] Password strength minimum: 8 karakter, kombinasi huruf besar/kecil, angka, dan simbol
- [ ] Password disimpan menggunakan **bcrypt** (cost factor ≥12) atau **Argon2id**
- [ ] Tidak ada plaintext password di database, log, atau kode
- [ ] Rate limit login: maksimal **5 kali gagal** dalam 15 menit → akun dikunci sementara (15 menit)
- [ ] Login gagal tidak membocorkan apakah username atau password yang salah
- [ ] Logout menginvalidasi token/session sepenuhnya di sisi server

### 2.2 JWT dan Cookie

- [ ] JWT disimpan hanya dalam **HttpOnly cookie** — bukan localStorage atau sessionStorage
- [ ] Cookie atribut: `Secure; HttpOnly; SameSite=Strict`
- [ ] JWT payload tidak mengandung data sensitif (hanya user_id, role, exp)
- [ ] Token tidak pernah dicetak ke console, log, atau response body yang terekspos
- [ ] Refresh token dirotasi setiap penggunaan (rotation policy)
- [ ] JWT signing key disimpan di Cloudflare environment variable — tidak di kode atau repo
- [ ] Token expired menghasilkan HTTP 401 dengan pesan generik
- [ ] Tidak ada token aktif setelah logout (server-side invalidation)

### 2.3 Multi-Role Session

- [ ] User dengan multiple role dapat beralih role tanpa logout (role switcher)
- [ ] Peralihan role mengeluarkan token baru dengan role aktif yang benar
- [ ] Audit mencatat: login, logout, dan peralihan role beserta IP + User-Agent

## 3. Otorisasi RBAC

### 3.1 17 Role Matrix

| No | Role | Permission Utama | Catatan Keamanan |
|---|---|---|---|
| 1 | Super Admin | Semua permission teknis | Harus dibatasi ke 2 orang max; setiap aksi diaudit |
| 2 | Admin SSK | Kelola plugin, user, konfigurasi | Tidak otomatis punya akses data finance |
| 3 | Admin SIK | Operasional sistem, monitoring | Akses sesuai kebutuhan Tim SIK |
| 4 | Admin OPD | Kelola data unit kerja | Scope terbatas per unit |
| 5 | Admin Faskes | Kelola data faskes sendiri | Hanya faskes_id yang sesuai |
| 6 | Pegawai | Buat dan lihat pengajuan sendiri | Tidak bisa approve pengajuan sendiri |
| 7 | Pelaksana | Lihat tugas dan upload bukti | Hanya tugas yang dia ikuti |
| 8 | Atasan Langsung | Approval tahap 1 | Hanya bawahan di unit sendiri |
| 9 | Kabid | Approval tahap 2 / verifikasi teknis | Sesuai bidang |
| 10 | Sekretaris | Verifikasi administrasi | Sesuai kewenangan sekretariat |
| 11 | Kadis | Approval final / akses ringkasan pimpinan | Read-only untuk data detail |
| 12 | Kepala Faskes | Approval pengajuan faskes | Hanya faskes sendiri |
| 13 | Kepala TU Faskes | Verifikasi administrasi faskes | Hanya faskes sendiri |
| 14 | Keuangan | Validasi biaya SPPD | Scope: dinas_all atau faskes_own |
| 15 | Operator Surat | Generate PDF, upload dokumen final | Tidak mengubah substansi yang sudah approved |
| 16 | Reviewer MMC | Review draft publikasi MMC | Tidak boleh melihat rincian biaya |
| 17 | Auditor | Read-only audit log dan arsip | Tidak bisa mengubah data apapun |

### 3.2 Checklist Permission Check per Route

- [ ] **Setiap** route handler memiliki permission check — tidak ada route yang melewati RBAC
- [ ] Permission check dilakukan di **server side** — client tidak dapat menentukan permission sendiri
- [ ] Default deny: jika permission tidak terdaftar, akses ditolak dengan HTTP 403
- [ ] Tidak ada permission escalation: user biasa tidak bisa memperoleh permission lebih tinggi melalui manipulasi request
- [ ] Role tidak dikirim dari client sebagai sumber kebenaran — role selalu diambil dari session server
- [ ] Setiap endpoint `POST/PUT/PATCH/DELETE` memiliki permission yang berbeda dari `GET`
- [ ] Permission `*.manage` tidak mencakup permission `audit.read` secara otomatis

### 3.3 RBAC Test Wajib

```
[ ] User tanpa permission agenda.read tidak bisa membuka daftar agenda
[ ] User tanpa permission duty.create tidak bisa membuat ST
[ ] User tanpa permission duty.approve tidak bisa melakukan approval
[ ] User tanpa permission duty.generate_document tidak bisa generate PDF
[ ] User tanpa permission duty.verify_evidence tidak bisa verifikasi bukti
[ ] User tanpa permission archive.read tidak bisa membuka arsip
[ ] User tanpa permission audit.read tidak bisa melihat audit log
[ ] User tanpa permission mmc.review tidak bisa review draft MMC
[ ] Operator surat tidak bisa mengubah substansi pengajuan yang sudah approved
[ ] Reviewer MMC tidak bisa mengakses dokumen finance
```

### 3.4 Verifikasi No Permission Escalation

- [ ] Header manipulasi (X-Role, X-Permission) tidak mengubah akses
- [ ] Parameter role dalam query string tidak mempengaruhi permission check
- [ ] JWT yang dimanipulasi menghasilkan signature mismatch → 401
- [ ] Tidak ada endpoint yang memberikan token dengan permission lebih tinggi dari session asli

## 4. Otorisasi ABAC (Attribute-Based)

### 4.1 Atribut yang Digunakan

| Atribut | Sumber | Keterangan |
|---|---|---|
| `user_id` | JWT session | ID user yang sedang login |
| `role` | JWT session | Role aktif user |
| `unit_id` | User profile | Unit kerja user |
| `health_facility_id` | User profile | ID faskes (jika relevan) |
| `created_by` | Database record | Pembuat entitas |
| `participant_user_ids` | Database record | Daftar peserta ST |
| `approval_step` | Database record | Tahap approval saat ini |
| `approver_user_id` | Database record | Approver yang ditunjuk |
| `document_classification` | Database record | Klasifikasi data: public/internal/restricted/confidential/finance |
| `is_budgeted` | Database record | Apakah pengajuan memerlukan anggaran |
| `keuangan_scope` | User profile | `dinas_all` atau `faskes_own` |
| `status` | Database record | Status entitas dalam state machine |
| `archive_classification` | Database record | Klasifikasi arsip dokumen |

### 4.2 8 ABAC Rules Wajib

**Rule 1 — Pegawai hanya lihat data sendiri**
```
Allowed jika:
  user_id == record.created_by
  ATAU user_id IN record.participant_user_ids
  ATAU role IN [Admin SSK, Admin SIK, Super Admin]
Denied: user biasa mengakses pengajuan user lain
```

**Rule 2 — Faskes scope isolation**
```
Allowed jika:
  user.health_facility_id == record.health_facility_id
  ATAU user.health_facility_id IS NULL (level dinas)
  ATAU role IN [Admin SSK, Admin SIK, Super Admin]
Denied: Admin Faskes A mengakses data Faskes B
```

**Rule 3 — Approval step: hanya approver aktif yang bertindak**
```
Allowed jika:
  record.status == 'pending_approval'
  DAN record.current_step == approval_step untuk role user ini
  DAN user_id == record.approver_user_id (atau sesuai unit/jabatan)
Denied: approver tahap 3 tidak bisa approve saat masih di tahap 1
```

**Rule 4 — Finance scope: dinas_all vs faskes_own**
```
Keuangan dengan keuangan_scope == 'dinas_all':
  Bisa melihat SPPD dan bukti biaya semua unit/faskes

Keuangan dengan keuangan_scope == 'faskes_own':
  Hanya bisa melihat SPPD dan bukti biaya faskes sendiri

User non-keuangan: tidak bisa mengakses data klasifikasi 'finance'
```

**Rule 5 — Archive classification: restricted/finance/confidential untuk role tertentu**
```
'public'       → semua user login
'internal'     → semua pegawai sistem
'restricted'   → pemilik dokumen + atasan + operator surat + admin
'confidential' → admin + pimpinan (Kadis, Sekretaris)
'finance'      → Keuangan (sesuai scope) + Auditor + admin
```

**Rule 6 — Journal: hanya pemilik, atasan, admin**
```
Jurnal pegawai dapat diakses oleh:
  - Pemilik jurnal (user_id == jurnal.user_id)
  - Atasan Langsung dari unit yang sama
  - Admin SIK, Admin SSK, Super Admin
  - Auditor (read-only)
Denied: pegawai lain atau user luar unit
```

**Rule 7 — MMC: hanya Reviewer MMC dan admin**
```
Draft MMC dapat diakses untuk review oleh:
  - Reviewer MMC
  - Admin SSK, Admin SIK, Super Admin
Draft tidak boleh berisi data pasien, rincian biaya, atau dokumen restricted
Publikasi hanya setelah Reviewer MMC menyetujui secara eksplisit
```

**Rule 8 — Audit log: hanya Auditor dan admin**
```
Audit log (satusehat_audit_logs) hanya dapat dibaca oleh:
  - Auditor (read-only, tidak bisa filter bypass)
  - Admin SSK, Super Admin
Audit log tidak dapat diedit atau dihapus oleh siapapun
```

### 4.3 ABAC Test Wajib

```
[ ] User Faskes A tidak bisa melihat data Faskes B
[ ] Pegawai A tidak bisa membuka draft ST Pegawai B
[ ] Approver tahap 2 tidak bisa approve saat pengajuan masih di tahap 1
[ ] Keuangan faskes_own tidak bisa melihat SPPD unit dinas lain
[ ] User tanpa role Keuangan tidak bisa mengakses dokumen klasifikasi finance
[ ] Reviewer MMC tidak bisa membuka bukti biaya
[ ] Auditor tidak bisa mengubah data apapun — hanya read-only
[ ] Pegawai tidak bisa melihat jurnal pegawai lain
[ ] Dokumen restricted hanya dapat diunduh oleh pemilik + atasan + admin
[ ] Dashboard tidak memperlihatkan data detail lintas unit kepada viewer umum
```

## 5. Keamanan Data Input

### 5.1 SQL Injection Prevention

- [ ] **Parameterized queries wajib** di semua query D1 — tidak ada string interpolation langsung
- [ ] Input dari user tidak pernah langsung dimasukkan ke raw SQL
- [ ] ORM/query builder AWCMS-Micro digunakan konsisten
- [ ] Filter kolom menggunakan allowlist (hanya field yang diizinkan)
- [ ] Sort parameter divalidasi terhadap allowlist field

**Test SQL Injection:**
```
Input uji pada search/filter: ' OR '1'='1 | "; DROP TABLE users; -- | admin'--
Expected: query tidak error terbuka, data tidak bocor, response aman
```

### 5.2 XSS Prevention

- [ ] Semua output ke HTML di-escape menggunakan library sanitizer
- [ ] Input rich text/HTML disanitasi sebelum disimpan (allowlist tag: bold, italic, list, link)
- [ ] Preview dokumen tidak menjalankan script dari konten
- [ ] Draft MMC tidak menyimpan tag `<script>` atau event handler inline
- [ ] CSP (Content Security Policy) header terpasang:
  ```
  Content-Security-Policy: default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; frame-ancestors 'none'
  ```

**Test XSS:**
```
Input uji: <script>alert('xss')</script> | <img src=x onerror=alert('xss')>
Expected: script tidak berjalan, output di-escape atau ditolak
```

### 5.3 File Upload Validation

| Jenis Upload | Tipe File Diizinkan | Batas Ukuran |
|---|---|---|
| Dokumen final signed | PDF saja | 10 MB |
| Lampiran agenda | PDF, DOC, DOCX, XLS, XLSX, JPG, PNG | 10 MB per file |
| Bukti foto | JPG, JPEG, PNG | 5 MB per file |
| Bukti dokumen | PDF, JPG, PNG | 10 MB per file |
| Bukti biaya | PDF, JPG, PNG | 10 MB per file |
| Batch bukti tugas | — | 50 MB total per batch |
| Asset MMC | JPG, JPEG, PNG | 5 MB per file |

- [ ] Validasi MIME type dari header file (tidak hanya extension)
- [ ] Validasi extension setelah normalisasi nama file
- [ ] Tolak file double extension: `bukti.pdf.exe`, `doc.php.pdf`
- [ ] Normalisasi nama file — strip path traversal: `../../secret.pdf` → ditolak
- [ ] Tolak file executable: `.exe`, `.sh`, `.php`, `.js`, `.html`, `.py`
- [ ] Semua file disimpan di R2 private bucket — tidak pernah public by default
- [ ] Nama file disimpan sebagai UUID + extension yang sudah divalidasi di R2

### 5.4 Path Traversal Prevention

- [ ] Nama file tidak pernah digunakan langsung sebagai path R2
- [ ] R2 key menggunakan format: `{plugin}/{entity_id}/{uuid}.{ext}`
- [ ] Download menggunakan signed URL — tidak ada direct file path di response

## 6. Keamanan Penyimpanan

### 6.1 Cloudflare D1 (Database)

- [ ] D1 menggunakan enkripsi at-rest secara native (Cloudflare managed)
- [ ] Tidak ada data plaintext password di tabel users
- [ ] Field classification (`public`/`internal`/`restricted`/`confidential`/`finance`) ada di setiap tabel dokumen
- [ ] Field `file_hash` (SHA-256) ada di setiap record dokumen final
- [ ] Field audit timestamp (`created_at`, `updated_at`) ada di setiap tabel

### 6.2 Cloudflare R2 (File Storage)

- [ ] R2 bucket bersifat **private** — tidak ada public access policy
- [ ] Download file menggunakan **pre-signed URL** dengan TTL maksimal 15 menit
- [ ] Server-side encryption aktif (R2 default)
- [ ] File path di R2 tidak dapat ditebak (menggunakan UUID)
- [ ] Setiap download dokumen restricted/finance/confidential dicatat ke audit log

### 6.3 Cloudflare KV (Cache)

- [ ] Dashboard KV cache menggunakan TTL **15 menit**
- [ ] Nilai sensitif di KV menggunakan enkripsi pada layer aplikasi
- [ ] Tidak ada data PII tersimpan langsung di KV tanpa enkripsi
- [ ] KV key tidak mengandung data user yang dapat ditebak

### 6.4 Secret Management

- [ ] Tidak ada secret (API key, token, password) dalam kode atau repository
- [ ] Semua secret disimpan di **Cloudflare environment variables** (`wrangler secret`)
- [ ] `.env` tidak pernah di-commit — hanya `.env.example` dengan placeholder
- [ ] Rotasi secret dilakukan sebelum go-live production
- [ ] AI coding agent tidak boleh mencetak nilai secret — hanya nama variabel

## 7. Keamanan Jaringan

### 7.1 Konfigurasi Wajib

- [ ] **HTTPS only** — HTTP redirect ke HTTPS (HTTP 301)
- [ ] HSTS header: `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload`
- [ ] CSP header terpasang (lihat bagian 5.2)
- [ ] `X-Content-Type-Options: nosniff`
- [ ] `X-Frame-Options: DENY`
- [ ] `Referrer-Policy: strict-origin-when-cross-origin`
- [ ] `Permissions-Policy: camera=(), microphone=(), geolocation=()`

### 7.2 CORS

- [ ] CORS hanya mengizinkan domain resmi sistem (allowlist):
  ```
  https://satusehatkobar.dinkes-kobar.go.id
  https://staging.satusehatkobar.dinkes-kobar.go.id
  ```
- [ ] Wildcard `*` tidak diizinkan untuk CORS origin
- [ ] Method yang diizinkan: `GET, POST, PUT, PATCH, DELETE, OPTIONS`
- [ ] Credential CORS: `Access-Control-Allow-Credentials: true` hanya untuk domain yang terdaftar

### 7.3 Rate Limiting

- [ ] Rate limit per user: **100 request/menit**
- [ ] Rate limit per IP (unauthenticated): **30 request/menit**
- [ ] Endpoint upload file: **10 upload/menit per user**
- [ ] Endpoint generate PDF: **5 generate/menit per user**
- [ ] Login endpoint: **5 kali gagal/15 menit → temporary lock**
- [ ] Rate limit exceeded menghasilkan HTTP 429 dengan `Retry-After` header

### 8.3 Security Monitoring Alerts

- [ ] HTTP 403 (permission denied) > 20/menit per user → alert + investigasi
- [ ] Login gagal > 5x dalam 15 menit (per user atau IP) → alert + temporary lock
- [ ] Upload file gagal validasi > 5x dari IP yang sama → flag investigasi
- [ ] D1 storage > 80% kapasitas → alert kapasitas
- [ ] R2 storage > 80% kapasitas → alert kapasitas

- [ ] Auditor hanya memiliki akses read-only ke audit log
- [ ] Export audit log dibatasi untuk role Auditor dan Admin SSK
- [ ] Audit log tidak menyimpan nilai token, password, atau secret
- [ ] Audit log retention: 2 tahun aktif, 3 tahun inaktif → prosedur musnah formal

## 9. Privasi Data PII

### 9.1 Field yang Mengandung PII

| Entitas | Field PII | Klasifikasi | Penanganan |
|---|---|---|---|
| User profile | nama, NIP, jabatan, email, unit_id | internal | Akses sesuai role |
| ST/SPPD | nama_pemohon, NIP_pemohon (snapshot) | restricted | Snapshot untuk kebutuhan dokumen |
| SPPD | nama_peserta, NIP (snapshot) | finance | Hanya role finance |
| Bukti tugas | foto aktivitas (mungkin ada wajah) | restricted | Peringatan upload |
| Audit log | actor_name_snapshot, ip_address | audit | Hanya Auditor/Admin |
| Jurnal pegawai | isi jurnal, user_id | restricted | Hanya pemilik + atasan + admin |

### 9.2 Kebijakan PII dalam Snapshot Dokumen

- [ ] Snapshot PII (nama, NIP, jabatan) dalam dokumen ST/SPPD: **diizinkan** untuk keperluan audit historis
- [ ] Snapshot mengikuti klasifikasi dokumen induk (restricted/finance)
- [ ] Snapshot tidak dapat diedit setelah dokumen final
- [ ] Data pasien/klinis tidak boleh masuk dalam field apapun di sistem ini

### 9.3 Bukti Foto dan PII

- [ ] Saat upload bukti foto, sistem menampilkan **peringatan**: "Pastikan foto tidak mengandung data klinis pasien atau informasi sensitif yang tidak diperlukan"
- [ ] Klasifikasi bukti foto default: `restricted`
- [ ] Bukti foto dengan klasifikasi `finance` hanya dapat diakses role Keuangan dan Auditor
- [ ] Evidence PII scanning otomatis (AI-based): **Phase 2** — tidak di MVP

### 9.4 Tidak Ada Data Klinis/Pasien

- [ ] Field apapun di Agenda, ST/SPPD, bukti tugas, jurnal, dan MMC tidak boleh menyimpan data klinis pasien
- [ ] Tidak ada NIK pasien, nomor rekam medis, diagnosa, atau data medis dalam sistem ini
- [ ] MMC hanya berisi informasi kegiatan — bukan data rekam medis

## 10. Kepatuhan Regulasi

### 10.1 PP No. 95/2018 SPBE

- [ ] Sistem menggunakan infrastruktur pemerintah/cloud resmi
- [ ] Audit trail tersedia untuk akuntabilitas
- [ ] Standar keamanan informasi diterapkan
- [ ] Integrasi dengan sistem pemerintah mengikuti standar SPBE (Phase 2)

### 10.2 UU Perlindungan Data Pribadi (UU No. 27/2022)

- [ ] Hanya data yang diperlukan yang dikumpulkan (data minimization)
- [ ] Data digunakan sesuai tujuan awal (purpose limitation)
- [ ] Tidak ada transfer data ke pihak ketiga tanpa kebijakan yang jelas
- [ ] Prosedur pelanggaran data (data breach) tersedia

### 10.3 Permenkes No. 4/2019 SPM

- [ ] 12 indikator SPM tersedia di dashboard
- [ ] Data SPM hanya terkait kegiatan operasional, bukan data rekam medis
- [ ] Laporan SPM dapat diakses oleh pimpinan yang berwenang

### 10.4 Kepatuhan Kearsipan

- [ ] Metadata arsip mencakup: nomor, tanggal, jenis, unit, klasifikasi, hash
- [ ] Retensi arsip mengikuti Jadwal Retensi Arsip (JRA) yang berlaku
- [ ] Dokumen final tidak dapat dihapus tanpa prosedur formal

## 11. Penetration Test Checklist (Pre-Launch)

### 11.1 OWASP Top 10 Verification

| OWASP Category | Test | Status |
|---|---|---|
| A01: Broken Access Control | IDOR test, ABAC bypass, privilege escalation | [ ] |
| A02: Cryptographic Failures | Plaintext data check, weak algorithm check | [ ] |
| A03: Injection | SQL injection, XSS, command injection | [ ] |
| A04: Insecure Design | Workflow bypass, state machine bypass | [ ] |
| A05: Security Misconfiguration | CORS, CSP, headers, debug endpoints | [ ] |
