## 2\. Prinsip Utama Integrasi

Setiap integrasi eksternal wajib mengikuti prinsip berikut:

1. **MVP first**  
   Integrasi eksternal penuh tidak boleh menghambat penyelesaian MVP.

2. **Governance before code**  
   Integrasi tidak boleh langsung dikodekan sebelum ada analisis kebutuhan, data, risiko, keamanan, dan pemilik sistem.

3. **Plugin-first**  
   Integrasi dibuat sebagai plugin, adapter, service connector, atau extension. Jangan menanam logika integrasi langsung ke core AWCMS-Micro.

4. **Secure by default**  
   Semua credential, token, secret, sertifikat, dan API key harus dikelola secara aman.

5. **Audit everything**  
   Setiap pengiriman, penerimaan, perubahan status, error, retry, dan sinkronisasi penting harus tercatat.

6. **Fail safely**  
   Jika integrasi gagal, sistem utama tetap dapat berjalan dengan fallback manual.

7. **No silent sync**  
   Sinkronisasi tidak boleh berlangsung diam-diam tanpa log, status, dan notifikasi yang memadai.

8. **Human approval for critical actions**  
   Pengiriman dokumen, publikasi, pembayaran, dan keputusan administratif tetap harus melalui user berwenang.

9. **Data minimization**  
   Kirim hanya data yang diperlukan untuk integrasi.

10. **No clinical data without governance**  
    Integrasi yang menyentuh data klinis pasien wajib memiliki kajian khusus, kontrol akses ketat, dan persetujuan resmi.

# 5\. Integration Readiness Gate

Integrasi hanya boleh dimulai bila semua syarat berikut terpenuhi.

## 5.1 Business Readiness

* Tujuan integrasi jelas.

* Pemilik proses bisnis jelas.

* Pemilik sistem eksternal jelas.

* PIC teknis jelas.

* PIC operasional jelas.

* Alur manual/fallback tersedia.

* Dampak ke SOP dianalisis.

* Dampak ke user dianalisis.

* Keputusan Product Owner tersedia.

* Integrasi masuk roadmap fase lanjutan.

## 5.2 Technical Readiness

* Dokumentasi API/sistem eksternal tersedia.

* Environment sandbox/staging tersedia bila ada.

* Mekanisme authentication jelas.

* Mekanisme authorization jelas.

* Format request/response jelas.

* Rate limit diketahui.

* Error code diketahui.

* Retry mechanism dirancang.

* Timeout dirancang.

* Logging dan audit dirancang.

* Fallback manual dirancang.

## 5.3 Security Readiness

* Data yang dikirim sudah dipetakan.

* Data sensitif sudah diklasifikasikan.

* Credential tidak masuk repository.

* Secret disimpan aman.

* Transport aman.

* Audit integrasi tersedia.

* Access control tersedia.

* Incident response tersedia.

* Data pasien tidak diproses tanpa governance khusus.

* Risiko privasi dianalisis.

## 5.4 Operational Readiness

* User tahu kapan integrasi digunakan.

* User tahu jika integrasi gagal.

* Admin tahu cara retry.

* Admin tahu cara fallback manual.

* Support tahu cara membaca log.

* SOP integrasi tersedia.

* UAT integrasi tersedia.

* Monitoring integrasi tersedia.

* Rollback tersedia.

* Release note tersedia.

# 8\. Standar Arsitektur Integrasi

## 8.1 Pola Plugin Connector

Setiap integrasi dibuat sebagai plugin connector atau adapter.

Contoh struktur:

packages/plugins/  
└── awcms-micro-integration-tte/  
    ├── README.md  
    ├── CHANGELOG.md  
    ├── migrations/  
    ├── routes/  
    ├── handlers/  
    ├── services/  
    ├── connectors/  
    ├── validators/  
    ├── permissions/  
    ├── audit/  
    ├── tests/  
    └── docs/

## 8.2 Komponen Integrasi

Setiap plugin integrasi minimal memiliki:

1. Connector service.

2. Configuration schema.

3. Credential reference.

4. Request validator.

5. Response parser.

6. Error handler.

7. Retry policy.

8. Audit event.

9. Sync log.

10. Admin UI status.

11. Health check.

12. Documentation.

13. Test.

14. Fallback manual instruction.

15. Changelog.

---

## 8.3 Larangan Arsitektur Integrasi

Tidak boleh:

1. Menyimpan credential hardcoded.

2. Menyimpan secret di repository.

3. Mengirim data tanpa audit.

4. Mengirim semua data tanpa filter.

5. Membuat integrasi langsung di core.

6. Membuat sinkronisasi tanpa status.

7. Membuat retry tanpa batas.

8. Menyembunyikan error integrasi.

9. Membuka endpoint webhook tanpa authentication/verification.

10. Memproses data pasien tanpa governance khusus.
